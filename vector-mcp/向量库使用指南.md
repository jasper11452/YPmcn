# YPmcn 向量库使用指南

## 1. 概述

`vector-mcp/` 是一个独立的 **MCP (Model Context Protocol) Server**，通过 stdio JSON-RPC 通信。核心功能：**将达人的内容/成长标签向量化，支持基于语义相似度的达人搜索与匹配**。

### 架构总览

```
┌──────────────────────────────────────────────────────────────────┐
│                         向量库使用指南                              │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  数据源 (MySQL)               向量存储 & 搜索                     │
│  ┌──────────────┐            ┌──────────────────────────┐       │
│  │ xhs_creator_ │   sync    │  Qdrant (仅内存实现)      │       │
│  │ accounts     │ ────────→ │  ┌────────────────────┐  │       │
│  │ dy_creator_  │  提取标签  │  │ FakeQdrantClient   │  │       │
│  │ accounts     │  embed    │  │ - cosine 搜索       │  │       │
│  └──────────────┘            │  │ - BM25 稀疏搜索     │  │       │
│                              │  │ - in-memory store   │  │       │
│                              │  └────────────────────┘  │       │
│                              └──────────────────────────┘       │
│                                                                  │
│  ┌─ 搜索流程 ──────────────────────────────────────────────┐    │
│  │ 1. Query 归一化 → 2. Embedding → 3. 混合检索           │    │
│  │    (过滤品牌/数值/KPI)      (SiliconFlow / Fake)         │    │
│  │                                                         │    │
│  │ 4. RRF 融合 → 5. 负向过滤 → 6. 去重 → 7. Rerank → 输出 │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 两种运行模式

| 维度 | Fake 模式（默认） | Real 模式 (`VECTOR_MCP_MODE=real`) |
|------|------------------|-----------------------------------|
| **向量存储** | 内存 `FakeQdrantClient`，内置 60+ 条假数据 | 同样用 `FakeQdrantClient`，但数据从 MySQL 拉取 |
| **Embedding** | 确定性 FNV hash（128 维） | SiliconFlow `Qwen/Qwen3-Embedding-8B`（1024 维） |
| **Reranker** | Jaccard 词重叠度 | SiliconFlow `BAAI/bge-reranker-v2-m3` |
| **MySQL** | 不使用 | 连接 `xhs_creator_accounts` / `dy_creator_accounts` |
| **适用场景** | 本地开发、测试、演示 | 生产环境，真实数据搜索 |

## 2. 快速开始

### 2.1 Fake 模式（零配置）

直接启动即可，内置 60+ 条模拟达人数据，覆盖双平台、多垂类：

```bash
cd vector-mcp/
npm run build
node dist/server.js
```

MCP 客户端配置（opencode.json）：

```json
{
  "mcp": {
    "vector": {
      "type": "stdio",
      "command": "node",
      "args": ["vector-mcp/dist/server.js"],
      "enabled": true
    }
  }
}
```

### 2.2 Real 模式（连接 MySQL + SiliconFlow）

```bash
export VECTOR_MCP_MODE=real
: "${SILICONFLOW_API_KEY:?请先设置 SILICONFLOW_API_KEY}"
export MYSQL_HOST="your-db-host"
export MYSQL_USER="your-user"
: "${MYSQL_PASSWORD:?请先设置 MYSQL_PASSWORD}"
export MYSQL_DATABASE="your-database"

node dist/server.js
```

## 3. MCP 工具参考

### 3.1 `sync_creator_tag_vectors` — 同步标签向量

从 MySQL 拉取达人标签，embedding 后写入向量库。

**参数：**

| 参数 | 类型 | 说明 |
|------|------|------|
| `sourceMappingPath` | string | 字段映射 JSON 路径 |
| `platform` | `"xhs"` \| `"dy"` | 平台 |
| `dryRun` | boolean | 仅校验不写入 |

**数据来源字段映射：**

- 小红书：`content_type_label`, `content_theme_label`, `industry_tag_label`, `xt_talent_type_label`, `grow_talent_type_label`, `talent_type_label`
- 抖音：`content_type_label`, `kol_persona_label`, `content_feature_label`, `content_tag`, `business_industry`, `pgy_blogger_type_label`, `grow_blogger_type_label`, `talent_type_label`

**同步逻辑：** 每个达人拆分为 2 个向量点——`content` 标签和 `grow` 标签。向量点 ID 格式：
```
{platform}:{platform_account_id}:{tag_type}:{vector_version}
```

### 3.2 `search_creator_tag_vectors` — 搜索匹配达人

核心搜索工具，完整链路如下。

**参数：**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `positiveRequirements` | string[] | ✅ | 正向匹配标签 |
| `negativeRequirements` | string[] | ✅ | 排除标签 |
| `platform` | `"xhs"` \| `"dy"` | ❌ | 平台过滤 |
| `limit` | number | ❌ | 最大返回数，默认 10 |

或传完整需求对象（自动提取语义字段）：

| 参数 | 类型 | 说明 |
|------|------|------|
| `content_requirements` | string[] | 内容方向要求 |
| `creator_type_requirements` | string[] | 达人类型要求 |
| `tone_requirements` | string[] | 调性要求 |
| `negative_requirements` | string[] | 排除项 |
| `brand` / `project_name` / `product` | string | 品牌信息（自动过滤，不参与搜索） |
| `budget_raw` | string | 预算（自动过滤） |
| `geo` | string | 地理标签（如 "沈阳"） |

**响应结构：**

```json
{
  "success": true,
  "data": {
    "matches": [
      {
        "platform": "xhs",
        "platform_account_id": "xhs_mom_001",
        "tag_type": "content+grow",
        "matched_tags": ["母婴", "亲子", "育儿"],
        "raw_score": 0.85,
        "rerank_score": 0.92,
        "reason": "vector_similarity+rerank(0.920)"
      }
    ],
    "below_threshold": [],
    "positive_query": "母婴 亲子 育儿",
    "negative_terms": ["搞笑"],
    "total_candidates": 45,
    "deduped_candidates": 30,
    "platform_filter": null
  },
  "trace_id": "uuid"
}
```

### 3.3 `health_check_vector_store` — 健康检查

## 4. 搜索管线详解

```
输入需求 Brief
    │
    ▼
┌─────────────────────────────┐
│ 1. Query 归一化              │  query/normalize.ts
│   - 提取语义字段             │  extractVectorQuery()
│   - 过滤品牌/预算/KPI 数值   │  → positiveTerms + negativeTerms
│   - 识别负向前缀词           │
└──────────┬──────────────────┘
           ▼
┌─────────────────────────────┐
│ 2. Geo 预过滤（可选）        │  在内存中按 raw_tags
│   - 筛选包含地理标签的达人    │  做包含匹配
│   - 缩小搜索池              │
└──────────┬──────────────────┘
           ▼
┌─────────────────────────────┐
│ 3. Embedding                │  Fake: FNV hash × 128
│   - 查询文本 → 向量         │  Real: Qwen3-Embedding-8B × 1024
│   - 并行发起稠密 + 稀疏搜索   │
└──────────┬──────────────────┘
           ▼
     ┌─────────────────┐
     │ Dense Search    │   Cosine 相似度
     │ (向量搜索)       │   limit=1000, threshold=-1
     └────────┬────────┘
              │
     ┌────────┴────────┐
     │ BM25 Search     │   基于标签文本的稀疏检索
     │ (关键词搜索)     │   k1=1.2, b=0.75
     └────────┬────────┘
              │
           ▼
┌─────────────────────────────┐
│ 4. RRF 融合                  │  Reciprocal Rank Fusion
│   - k = 60                  │  score = Σ 1/(k + rank + 1)
│   - 稠密 + 稀疏结果合并排序   │
└──────────┬──────────────────┘
           ▼
┌─────────────────────────────┐
│ 5. 平台过滤 + 负向匹配        │
│   - platform 预过滤          │
│   - 检测候选标签是否命中负向词 │
└──────────┬──────────────────┘
           ▼
┌─────────────────────────────┐
│ 6. 去重合并                   │  deduplicateByAccountId()
│   - 同一达人的 content/grow   │
│     标签合并                 │
│   - 保留最高分，合并 raw_tags │
└──────────┬──────────────────┘
           ▼
┌─────────────────────────────┐
│ 7. Rerank（仅 safe 候选）    │  Fake: Jaccard 重叠度
│   - safe 候选取 top 50       │  Real: bge-reranker-v2-m3
│   - 按 rerank_score 降序排列  │
└──────────┬──────────────────┘
           ▼
┌─────────────────────────────┐
│ 8. 阈值过滤 + 拼接输出        │
│   - rerank < 0.05 → below_threshold
│   - safe 按 rerank 降序      │
│   - penalized 追加到末尾     │
│   - 按 limit 截断            │
└─────────────────────────────┘
```

## 7. 与主系统集成

`vector-mcp` 是主链路（12 个 SSE MCP 生产工具）的**补充能力**：

```
主链路 (生产 SSE MCP: https://mcp.eshypdata.com/sse):
  validate_requirement → search_creators → rank_mcns → create_with_distributions → ...

补充分支 (本地 stdio MCP):
  vector-mcp/search_creator_tag_vectors  →  标签向量相似度匹配
```

主系统通过 SSE 协议连接生产 MCP Server，提供完整的需求管理、达人搜索、MCN 排序、项目分发流程。`vector-mcp` 提供独立的标签向量搜索能力，主要用于：
- 本地开发阶段快速验证搜索算法
- 作为独立能力被其他 Agent 调用

## 8. 常见问题

**Q: 为什么 Qdrant 客户端是 Fake 的？**  
接口 `QdrantClientLike` 已定义完整（`ensureCollection`, `upsert`, `search`），`FakeQdrantClient` 是完整的内存实现。真实 Qdrant 连接暂未完成，当前数据全在进程内存中。

**Q: Fake 模式的搜索精度如何？**  
FNV hash 生成的 128 维 fake embedding 具有**确定性**（同一文本永远生成同一向量），cosine 相似度可正确反映文本相似关系。适合开发测试，不适合生产精度要求。

**Q: Bailian（阿里云百炼）Provider 是否可用？**  
接口已定义，但 `embed()` 和 `rerank()` 均 throw `"not yet implemented"`。当前可用 Provider 只有 SiliconFlow。

**Q: 如何处理不同版本的向量？**  
向量点 ID 中包含 `vector_version`（当前为 `v1`），版本变更时会自动生成新 ID，新旧向量共存，不会互相覆盖。

**Q: RRF 中的 k=60 是什么意思？**  
防止极端高排名主导结果。k=60 时，排名第 1 贡献 `1/61 ≈ 0.016`，排名第 100 贡献 `1/160 ≈ 0.006`——差距足够区分但不会让第一名完全压制后续结果。
