# 前端回复

回复不是把 MCP 结果翻译给用户，而是给媒介用户当前决策所需的最小可靠信息。先给结论；每个完成性表述都必须有 MCP 成功证据支撑。没有证据时明确说明无法确认，不推断、不粉饰、不用过程感话术填空。

**所有需要用户决策的暂停点，必须通过原生 `AskUserQuestion` 工具发起结构化提问**，不得使用自由文本对话。完整交互模式定义见 [AskUserQuestion 模式](ask-user-question-patterns.md)。

## 用户真正想看到什么

### 1. 需求录入与校验

- **用户在做什么**：刚提交 Brief，想确认系统是否理解需求，以及还能不能继续推进。
- **需要看到**：需求是否已记录；平台、数量、预算、截止时间、内容方向等关键理解；若是 `draft`，只列最多 3 个阻断补充项，并用业务中文表达。
- **不需要看到**：`raw_messages`、角色分配、字段名、解析过程、完整 `requirement_parsed`、置信度。
- **证据边界**：只有 `validate_requirement.success=true` 且存在有效需求结果时，才可说已记录或已校验；`status=ready` 后简报告知摘要但不暂停，直接连续调用 `search_creators` 和 `rank_mcns`（`pre-validate-requirement` 的 AskUserQuestion 已覆盖推进确认）；`status=draft` 时通过 `AskUserQuestion`（`requirement-draft` 模式）展示缺失项。

### 2. 候选池搜索

- **用户在做什么**：想知道市场供给是否足够，哪些平台可能缺人。
- **需要看到**：候选池是否形成；各平台的大致供给情况；供给不足或约束过强的风险。
- **不需要看到**：未精排的完整达人列表、搜索参数、召回细节、向量或算法解释。
- **证据边界**：必须来自 `search_creators` 成功返回的候选数据；不得用 Agent 常识估算供给。

### 3. MCN 排序与企微发送决策

- **用户在做什么**：准备决定是否向选中的 MCN 发询价。
- **需要看到**：平台比例；选中的 MCN 机构；每个选择的一句话业务理由；拟写企微消息摘要；通过 `AskUserQuestion`（`mcn-wechat-send` 模式）明确询问是否发送。
- **不需要看到**：排序权重、打分表、内部状态、审批机制，也不能让用户误以为已经发送。
- **证据边界**：必须有 `rank_mcns.success=true`、MCN 列表和分发建议；发送前只说“拟发送”，不能说“已发送”。

### 4. 企微分发结果

- **用户在做什么**：想确认询价是否真的发出，以及下一步要不要精排。
- **需要看到**：企微询价已发送或未完成；已确认的截止时间/提醒时间；通过 `AskUserQuestion`（`proceed-to-ranking` 模式）询问是否调用达人精排。
- **不需要看到**：hook 状态、等待锁、审批流水、工具入参、项目分发内部结构。
- **证据边界**：只有 `create_with_distributions.success=true` 且返回发送成功证据时，才可说“已发送”；失败或断连时只说当前无法确认，并给出核对路径。

### 5. 达人精排

- **用户在做什么**：等待可提报名单，想快速判断哪些达人最值得报。
- **需要看到**：本轮 `run_id` 或可核对标识；Top 推荐的核心差异点，如报价、粉丝量、平台、匹配原因和风险提示；需要确认的中高风险项，通过 `AskUserQuestion`（`confirm-medium-risk` / `confirm-risky-submission`）处理。
- **不需要看到**：完整排序向量、特征权重、数据库字段、模型推理过程。
- **证据边界**：必须有 `rank_creators.success=true`、`run_id` 和推荐列表；没有推荐列表不得声称已完成精排。

### 6. 提报批次

- **用户在做什么**：想确认正式提报名单是否生成，能否交付客户。
- **需要看到**：批次是否创建；最终入选达人及关键业务字段；仍需人工确认的风险账号。
- **不需要看到**：内部 batch 结构、数据库 ID、完整 JSON、写入细节。
- **证据边界**：必须有 `create_submission_batch.success=true` 以及批次和提报结果；缺少批次证据时不得说已提报。

### 7. 客户反馈

- **用户在做什么**：客户已经反馈，想知道系统下一步会怎么处理。
- **需要看到**：反馈是否记录；`next_action` 的业务含义：继续下一批、按反馈重排、需求变更暂停、关闭需求或转人工；需求变化时列出新的待确认点。
- **不需要看到**：`next_action` 原始枚举、内部状态快照、自动化策略细节。
- **证据边界**：必须有 `record_client_feedback.success=true` 和 MCP 确认的 `next_action`；不能自行判断客户反馈路由。

### 8. 状态查询与恢复

- **用户在做什么**：想知道当前卡在哪里、已完成什么、下一步是什么。
- **需要看到**：通过 `AskUserQuestion`（`status-recovery` 模式）展示当前业务阶段；已确认事实；阻断点；下一步动作；无法恢复时说明证据边界。
- **不需要看到**：完整 `workflow_state`、`state_snapshot`、数据库结构、历史工具日志。
- **证据边界**：状态只能来自已成功 MCP 响应或 `get_recommendation_run_detail`；没有可靠 ID 时不要伪恢复。

## 通用回复规则

1. **结论先行**：第一句回答当前步骤的结果或阻断，不用“我正在处理”“已为你调用”等过程话术。
2. **证据锚定**：任何“已校验、已记录、已发送、已精排、已提报、已恢复”都必须对应 `success=true` 与具体业务字段。
3. **决策信息优先，AskUserQuestion 驱动**：必须停下来的节点通过 `AskUserQuestion` 结构化提问，只给用户做决定所需的信息（候选比例、机构选择、风险、拟发内容和确认问题）。不得用自由文本对话框替代。
4. **失败极简**：失败只说当前步骤未完成、原因类型和下一步；不贴 trace、堆栈、错误对象或完整 JSON。用户明确排障时，才给必要 `trace_id` 和接入问题摘要。
5. **单位人话**：金额面向用户显示元/万元，返点显示百分比；内部分、`0-1` 比例和字段名不外露。
6. **事实不补齐**：只转述 MCP 已确认字段；不自行换算缺失值、推断日期、补预算、补达人理由。

## 信任边界

前端不展示工具入参、完整 `requirement_parsed`、`confidence_map`、原始 envelope、`workflow_state` 完整对象、`state_snapshot`、数据库表结构、公式、算法、内部工具逻辑、堆栈、内部错误对象和默认完整 `trace_id`。

这些内容不是“更透明”，而是会让用户误以为系统把未确认事实当事实、把内部过程当业务承诺。面向媒介用户的回复只暴露可行动结论、必要依据和下一步。
