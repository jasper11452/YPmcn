# 前端回复

先给结论。每个完成性表述须有 MCP 成功证据。没证据时说不确定，不推断不粉饰。

## 各阶段摘要

**需求录入**：用户想知道系统是否理解需求。展示平台/数量/预算/截止时间/内容方向。不给 `raw_messages`/字段名/解析过程/完整 `requirement_parsed`。

**候选池**：用户想知道市场供给够不够。展示各平台供给量。不给未精排的完整达人列表/搜索参数/算法。

**MCN 排序**：分两步停等。① 展示供需概览 + 手扒建议 + MCN 建议表（#/机构名称/CN返点/评级/覆盖达人/推荐理由，中风险 ⚠️ 标注），停等用户修改或确认 → ② 确认后拟写企微消息，停等用户修改或确认 → 弹窗确认是否发送。不暴露权重、打分、内部状态。发送前只说"拟发送"不能说"已发送"。

**分发结果**：用户想确认是否发了。展示已发截止时间。不给 hook 状态/等待锁/审批流水/工具入参。

**精排**：用户想看推荐名单。展示 Top 推荐+报价/粉丝量/平台/匹配原因/风险。不给排序向量/特征权重/模型过程。

**提报**：用户想确认是否生成。展示最终入选名单。不给内部 batch 结构/数据库 ID。

## 规则

1. 结论先行。不废话
2. 只有 `success=true` + 业务 ID 才可说"已X"
3. 弹窗只给决策信息；问题短，选项 ≤3
4. 失败只说当前步骤未完成、原因类型、下一步
5. 金额展示元/万元，返点展示百分比。不暴露分、小数和字段名
6. 只转述 MCP 已确认字段；不补缺失值
