# YPmcn Claude Code 协作建议

## 适用范围

- 正式契约仍以 `spec/manifest.json` 的 `contracts` 映射为入口，安全与数据完整性优先。
- 本文件用于提示推荐流程，不建立角色隔离、工具白名单或固定完成门槛。
- Claude Code 可以直接读取、修改、运行测试、调用 Bash、使用 Skill 和调用 MCP；Codex、OpenCode 及其包装器均为可选协作者。

## 推荐工作方式

1. 先明确 goal、相关路径、acceptance 和最小验证，再选择最短可行路径。
2. 小型或上下文明确的任务直接执行。边界化实现可交给 Codex，研究或独立复核可交给 OpenCode；委派没有收益时无需委派。
3. OpenCode 若具备所需工具，可以作为流程操作者；若它缺少 MCP 或写入能力，Claude Code 应直接调用 MCP 或改用可用工具继续，不因此报告 `BLOCKED`。
4. Codex 完成后按风险决定是否需要 OpenCode 独立验证。OpenCode `PASS` 是有价值的证据，但不是所有任务的固定完成条件。
5. 包装器失败、格式不符或媒介权限不足时，优先切换执行路径。只有任务本身存在无法绕开的必要条件时才报告真实阻塞。

## 实机与 MCP 流程建议

- 尽量遵循 Spec 的阶段顺序、使用实际返回的 ID，并以真实 MCP 结果作为成功证据。
- `PreToolUse` Hook 只给出阶段、字段、确认项和数据一致性提醒，不批准或拒绝调用；最终判断由当前任务上下文、Claude Code 正常权限机制和用户授权共同决定。
- 缺少非关键测试参数时可以采用清晰标注的合理默认值。例如 E2E 测试可默认达人数量为 10、把过期档期平移到近期，并为记录增加唯一测试标识。
- 优先使用隔离测试环境。生产环境、不可逆动作或真实外部业务写入尚未获得授权时，只询问一次关键确认；用户已经明确授权完整流程后，不在每个阶段重复确认。
- 不伪造成功、不发明下游 ID，也不使用 shell 或 curl 绕过声明的 provider/MCP 写接口。

## 验证建议

默认运行任务相关测试和 `git diff --check`，再按改动范围选用：

| 命令 | 适用场景 |
|---|---|
| `npm run test:fast` | YPmcn 日常运行逻辑 |
| `npm run test:openclaw` | Plugin、Skill、manifest 或 OpenClaw 适配 |
| `npm run test:headless` | 跨组件联调或明确要求的合并前验证 |
| `npm run verify:docs` | 人类文档同步检查 |
| `npm run verify:provider` | 生产 provider 独立只读检查 |

测试未运行应标记 `NOT RUN`，失败不得描述为通过。验证强度应与风险相称，不为满足固定链路而重复运行无关检查。

## 最终输出

优先报告结论、改动、已执行验证和剩余风险。仅在缺少必要权限、关键输入或外部条件且没有安全替代路径时使用 `BLOCKED`，并给出一个准确原因和继续所需条件。
