# 完整业务需求与测试 Checklist

## 0. 范围、证据边界与已确认决策

**范围。** 本清单合并 22 份《业务需求与测试用例.md》及《全量汇总.md》，覆盖 Brief/自然语言输入、达人与 MCN 资源、筛选推荐、数据可信度、供应商提报、企微闭环、审计权限和非功能验收。相同意图只保留一条 BR；原文中的录音编号仅作为证据来源，不作为新的功能分支。

**证据边界。** 仅采用上述文档中明确或反复讨论的业务意图；未确定的字段、阈值、接口、抓取范围、更新频次、算法和规模不写成既定事实。录音中的噪声、个人行政事务和未形成输入—处理—输出—人工动作的讨论不进入首版验收。数据结果必须能回溯到来源和当时版本。

**已确认决策（本版必须落实）。**
1. 账号类型与内容标签字段间为 OR；未提供的条件不虚构。
2. 流程为两阶段：先硬筛并判断资源充足性，再排序/推荐/人工调整。
3. 数据冲突保留所有来源；平台真实数据优先，冲突进入人工复核。
4. 企微为生成→预览→人工确认→发送；首版不自动拉群、加人、踢人。
5. 资源分层补充：分开统计 MCN、野生、去重总量；优先机构提报，不足时经人工确认补野生达人。
6. Brief 明确返点下限/区间时先作硬筛；通过硬筛后，返点作为可配置排序因素。
7. 三层数量分开保存：客户最终目标、机构提报数量、内部候选池数量；倍率可配置。

优先级：**P0**=首版阻塞/必须；**P1**=首版应有；**待确认**=证据不足，确认前不得作为固定验收规则。

## 1. 分阶段业务需求 Checklist

### P0 治理与数据模型

- [ ] **BR-001｜P0** 建立需求、达人、MCN/机构、标签、提报、推荐批次、企微消息及人工复核记录的关联标识；验收：同一 Brief 可追溯到筛选条件、候选来源、调整记录和发送/回收状态，重复运行不覆盖历史。
- [ ] **BR-002｜P0** 保留原始输入、解析结果、人工修改值和确认状态；验收：未提及字段为空/待补充，不能被模型臆填；修改前后值和操作者可查看。
- [ ] **BR-003｜P0** 达人记录至少区分名称/标识、平台、账号类型、内容标签、数据、所属机构及来源；验收：字段缺失可标记未知，不以未知当作不符合。
- [ ] **BR-004｜P0** 对同一账号的多来源记录做可追溯合并而不丢源；验收：每个字段能看到来源、采集时间和原值，冲突进入复核队列。
- [ ] **BR-005｜P0** 所有硬筛、排序、人工调整、复核、发送和状态变更写入审计事件；验收：事件含时间、操作者、批次、动作和前后值，能按项目检索。

### P1 Brief 解析

- [ ] **BR-006｜P0** 支持自然语言、原始 Brief 或语音转写文本作为输入并解析结构化需求；验收：能识别的字段展示为结构化值，不能识别的保留原文并标记待确认。
- [ ] **BR-007｜P0** 建立 Brief 与解析字段、筛选条件、推荐结果及企微模板的强映射；验收：字段修改后下游读取最新已确认值，未要求的字段不凭空出现在结果中。
- [ ] **BR-008｜P0** 区分必填、建议填写、软性要求和不确定项；验收：缺少已配置必填项时追问或阻止进入对应阶段，选填缺失不被一律阻止。
- [ ] **BR-009｜P1** 结构化保存平台、项目/内容诉求、账号类型、内容标签、数据条件、预算/报价、返点、截止时间及数量（仅在输入提及时）；验收：每项保留原表达与规范化值，模糊边界要求人工确认。
- [ ] **BR-010｜P1** 支持媒介修改解析结果并确认后继续；验收：确认版本可回看，重跑使用确认值；方向改变与原需求补充不得无提示地混为同一批次。

### P2 标签与资源

- [ ] **BR-011｜P0** 账号类型与内容标签采用 OR 关系；验收：只填任一字段可命中对应条件；两者都填时命中任一即可，不因缺少另一字段误筛除。
- [ ] **BR-012｜P1** 保留一级/二级标签、内容标签和关键词的来源及匹配解释；验收：标签未配置或来源不足时标记无法判定，不宣称完成相似匹配。
- [ ] **BR-013｜P0** 资源池按达人/机构关系及平台等可用字段查询，并统计符合硬条件的供给；验收：结果显示命中/未命中的条件和数据缺失项。
- [ ] **BR-014｜P0** 分别保存 MCN 数量、野生达人数量和去重后总量；验收：同一达人跨机构/来源不重复计入总量，分层数相加关系可解释。

### P3 硬筛和三层数量

- [ ] **BR-015｜P0** 先执行硬筛，再进行资源充足性判断；验收：硬条件未通过的账号不进入排序池，系统输出筛后供给数及判断所用需求数。
- [ ] **BR-016｜P0** 平台、明确数据门槛、预算/报价边界、返点下限/区间等明确硬条件按原意执行，不默认浮动；验收：下限按不低于、区间按区间，单值无法确定等值/上下限时追问。
- [ ] **BR-017｜P0** 分开保存客户最终目标数、机构提报数、内部候选池数，并保留配置的补充倍率；验收：三者在界面、接口/导出和审计中不可混用，变更倍率不改写客户目标。
- [ ] **BR-018｜P0** 资源充足性判断支持不足提示并进入分层补充流程；验收：充足时进入排序，不足时展示缺口、各层数量和人工确认入口，不自动扩散。
- [ ] **BR-019｜P1** 筛选结果可回填、复跑和查看筛选依据；验收：同一输入与数据版本可复现，人工排除/保留有原因。

### P4 排序推荐与人工确认

- [ ] **BR-020｜P0** 硬筛通过后按可配置规则排序/推荐，返点仅作为已通过硬筛后的可配置因素；验收：开关/权重变化影响排序，不改变硬筛结果。
- [ ] **BR-021｜P1** 保留软性要求、标签相关度和数据完整性作为排序或人工参考，而非未经确认的硬条件；验收：结果展示因素和解释，缺数据不伪造高匹配。
- [ ] **BR-022｜P0** 推荐结果支持人工保留、删除、补充、调序、改数量和标注原因；验收：人工调整后生成新版本，原始自动结果仍可恢复。
- [ ] **BR-023｜P0** 在关键节点暂停等待人工确认：解析确认、资源不足补充、推荐调整、企微发送；验收：未确认不能越过该节点或发送。

### P5 数据采集与可信度

- [ ] **BR-024｜P0** 支持已有真实数据先行测试，并记录采集时间、来源、字段状态和数据版本；验收：候选结果可定位到具体数据快照。
- [ ] **BR-025｜P0** 多来源数据全部保留；平台真实数据优先，冲突标记并进入人工复核；验收：优先展示平台值但不删除其他值，复核结论和依据可追踪。
- [ ] **BR-026｜P1** 数据缺失、过期、格式异常或来源不可验证时显式标记；验收：不把缺失当零、不把未验证当真实，筛选和排序按配置处理并提示影响。
- [ ] **BR-027｜待确认** 明确抓取/更新的范围、频次、优先级和规模测试计划；验收：确认前只验收来源/版本留痕，不验收具体频次或规模。（影响采集调度、成本、容量和结果时效。）

### P6 MCN/野生补充

- [ ] **BR-028｜P0** 资源不足时优先发起机构提报并单独统计提报、有效提报、选中等状态；验收：机构提报数量不与客户目标或候选池数量混淆。
- [ ] **BR-029｜P0** 机构提报不足时，经人工确认后补充野生达人；验收：系统展示缺口、来源层和确认人；未确认不自动扩散或纳入最终推荐。
- [ ] **BR-030｜P1** 供应商可确认、删除或补充账号并回填结果；验收：每项变更有来源、时间和状态，内部可复核完整提报链路。
- [ ] **BR-031｜待确认** 确认机构提报时限、催收周期、有效提报口径和周/月统计口径；验收：确认前只验收状态留痕，不固定时间或统计阈值。（影响催收、报表和供应商评价。）

### P7 企微与供应商闭环

- [ ] **BR-032｜P0** 根据已确认 Brief 生成企微询价/需求消息和表单内容；验收：模板字段与 Brief 一致，缺失项留空/提示，不把待确认内容写成事实。
- [ ] **BR-033｜P0** 企微严格执行生成→预览→人工确认→发送；验收：预览可编辑，未确认不能发送，发送成功/失败及回执留痕。
- [ ] **BR-034｜P0** 首版不自动拉群、加人、踢人；验收：系统无自动执行这些动作；需要操作时提供人工路径和结果记录。
- [ ] **BR-035｜P1** 记录群消息、表单/链接、供应商响应、提交、回收和催收状态；验收：按项目、供应商和批次可追踪端到端状态。
- [ ] **BR-036｜待确认** 确认企微账号、群类型、机器人授权、模板字段和失败重试规则；验收：确认前仅验收人工发送和失败可见，不固定接口或权限。（影响集成、权限和运营流程。）

### P8 审计权限与非功能

- [ ] **BR-037｜P0** 按角色限制查看原始需求、资源数据、推荐调整、发送和配置操作；验收：无权角色无法读取或修改，越权尝试留痕。
- [ ] **BR-038｜P0** 支持人工复核队列（数据冲突、缺失、资源不足、边界不清）及处理结果；验收：每条有状态、责任人、结论、依据和时间。
- [ ] **BR-039｜P1** 支持按项目/批次导出结构化需求、候选、来源、数量分层、调整和发送状态；验收：导出与页面版本一致且敏感字段按权限脱敏。
- [ ] **BR-040｜P1** 提供失败提示、可重试或人工接管，不静默丢数据；验收：解析、筛选、采集、发送任一步失败均能定位环节并保留已完成结果。
- [ ] **BR-041｜待确认** 确认本地/云端部署、并发、容量、性能和备份保留目标；验收：确认前先完成小范围真实流程，暂不设置未被证据支持的数值。（影响架构、成本、运维和验收环境。）

## 2. 测试 Checklist

每条 TC 均要求记录：执行人、时间、输入/数据版本、实际结果、证据链接或截图、缺陷编号。除特别说明，均以测试环境和脱敏真实样本执行。

### 正常路径

- [ ] **TC-001｜BR-006,007,008** 前置：有含平台、数量、返点、标签和内容要求的自然语言 Brief。操作：解析、查看、补充缺失项并确认。预期：输出结构化字段；原文、规范化值和不确定项可追溯；确认后才进入筛选。
- [ ] **TC-002｜BR-009,016** 前置：Brief 含返点下限、预算区间和明确平台。操作：执行硬筛。预期：按下限/区间及平台筛选，不自动扩大范围；通过后返点可作为排序因素。
- [ ] **TC-003｜BR-011** 前置：资源记录分别有账号类型、内容标签。操作：分别只填类型、只填标签、同时填写两者筛选。预期：OR 逻辑正确；未填字段不被虚构为限制。
- [ ] **TC-004｜BR-015,018** 前置：资源池含足量和不足量两种样本。操作：先硬筛并运行充足性判断。预期：先输出硬筛数量；足量进入排序，不足展示缺口并停在人工确认。
- [ ] **TC-005｜BR-017** 前置：配置客户目标、机构倍率和候选倍率。操作：创建项目并运行提报/候选计算。预期：三层数量分别保存、展示和导出，倍率变化不改客户目标。
- [ ] **TC-006｜BR-020,022,023** 前置：有多条硬筛通过结果。操作：配置排序因素，人工调序、删除、补充并确认。预期：自动结果与人工版本分离；排序解释、操作者和原因完整；未确认不可进入发送。
- [ ] **TC-007｜BR-024,025,026** 前置：同一达人有平台真实值、机构值和冲突值。操作：导入并查看候选。预期：所有来源保留，平台值优先展示，冲突进复核，缺失/过期状态可见。
- [ ] **TC-008｜BR-028,029,030** 前置：机构提报不足且有可补野生样本。操作：发起机构提报，确认不足，再人工确认补野生。预期：先机构后野生；各层和去重总量分开统计；每次确认可追溯。
- [ ] **TC-009｜BR-032,033,035** 前置：Brief 已确认、推荐已人工确认。操作：生成企微消息，预览编辑，人工确认发送并模拟回执。预期：消息与 Brief 映射一致；未确认不能发；发送、失败、回收状态可查。

### 边界与异常

- [ ] **TC-010｜BR-008,009** 前置：分别缺少平台、截止时间、数量或返点。操作：逐项提交。预期：按当前配置提示必填缺失并阻止相应阶段；不把选填一律升级为必填。
- [ ] **TC-011｜BR-016** 前置：输入单值“3万”、模糊“左右”、以及上限/下限/区间表达。操作：解析并筛选。预期：明确边界按原意执行；无法确定的单值/模糊词进入确认，不擅自转区间或浮动。
- [ ] **TC-012｜BR-011,013** 前置：账号类型命中但内容标签缺失，或内容标签命中但类型缺失。操作：筛选。预期：按 OR 命中；缺失数据标记为缺失，不误报为不符合。
- [ ] **TC-013｜BR-017,018** 前置：客户目标大于机构提报和候选池供给。操作：运行充足性判断及倍率变化。预期：显示各层缺口和去重数，进入人工补充，不自动改目标或自动拉取。
- [ ] **TC-014｜BR-025,026** 前置：来源冲突、重复账号、过期数据和异常格式并存。操作：导入并运行筛选。预期：不丢来源、不重复计数；平台值优先；冲突/异常进入复核；缺失不当零。
- [ ] **TC-015｜BR-023,033,034** 前置：解析未确认、补充未确认、消息未确认，且账号无群管理授权。操作：尝试继续和发送。预期：每个节点阻断；无自动拉群/加人/踢人；失败原因和人工接管路径可见。
- [ ] **TC-016｜BR-040** 前置：模拟解析、采集、筛选和企微发送分别失败。操作：逐一重试。预期：明确定位失败步骤，已完成数据保留；重试不产生重复提报或重复发送。

### 权限、审计与人工确认

- [ ] **TC-017｜BR-037** 前置：准备媒介、审核人、管理员和只读账号。操作：分别查看、修改、确认、发送和导出。预期：权限按角色生效；越权被拒并审计。
- [ ] **TC-018｜BR-002,004,005,038** 前置：修改解析字段、处理数据冲突、人工删除候选。操作：查看版本和复核队列。预期：原值、来源、前后值、操作者、时间、理由和复核结论完整。
- [ ] **TC-019｜BR-022,023** 前置：自动推荐已生成。操作：人工保留、删除、补充、调序并撤销一次。预期：形成可恢复的新版本；未确认版本不可被下游读取或发送。
- [ ] **TC-020｜BR-033,035** 前置：消息有未确认、发送失败和成功三种状态。操作：预览、确认、发送、查看回执。预期：状态顺序正确，失败可人工接管，成功消息与项目/批次关联。
- [ ] **TC-021｜BR-039,040** 前置：项目含多版本、敏感字段和失败记录。操作：按角色导出并检索。预期：只导出有权字段；版本与页面一致；失败和审计事件可检索。

## 3. 待确认清单（不重复已确认七项）

以下事项在证据中仍阻塞实现或验收；未确认前不得擅自写入固定规则：

- [ ] **CF-001** 必填字段的最终清单、缺失字段的追问/放弃交互。推荐默认：先以平台、项目/内容、截止时间、客户目标数作为候选必填，其余按项目配置；影响：Brief 解析、流程阻断和验收样例。
- [ ] **CF-002** 标签字典、层级、来源及相似度/关键词算法。推荐默认：首版仅验收已配置标签的可解释匹配，未配置则人工处理；影响：标签模型、排序和数据准备。
- [ ] **CF-003** 需求变更时重跑、补充或新建批次的判定。推荐默认：方向/硬条件改变新建版本，软性补充沿用批次但保留版本；影响：幂等、提报去重和审计。
- [ ] **CF-004** 数据更新频次、采集范围、来源优先级（平台真实数据优先已确认）和规模测试安排。推荐默认：先用小规模真实快照验证闭环，再评估扩大；影响：采集、成本、容量和时效。

## 4. 风险与不纳入首版

- **外部平台抓取/RPA：** 平台规则、登录、验证码、字段变化、频控和合规风险会影响稳定性与可持续性；首版不把具体抓取接口、频次、规模或成功率当作已承诺验收项，先以可追溯真实数据和人工补录完成闭环。
- **ASR 证据限制：** 录音转写可能存在术语、数字、说话人和上下文误识别；ASR 只能作为需求输入证据，不能替代 Brief 确认、平台真实数据或人工复核；所有不确定字段必须可见并确认。
- **入职行政录音：** 新员工工位、电脑、门禁、会议、就餐、出行等行政内容不纳入本业务系统范围；仅保留其中涉及真实业务规则、样本和联调安排的证据。
- **不纳入首版自动动作：** 自动拉群、加人、踢人，以及未经人工确认的资源扩散、企微发送均不纳入；不把讨论中的阈值、倍率、接口和算法当作默认事实。
