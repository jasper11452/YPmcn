# 高效联调测试指南

> 当前基线：3.4.25。本指南区分“本地验证通过”“远端契约一致”和“真实业务链跑通”；三者不能互相代替。

## 先看结论

现在可以稳定验证 Plugin、4 个 Native Hook、Skill 装载和本地契约；但不能把这些结果写成“全链路已上线”。远端 Provider 目前仍有两项硬契约差异：

1. create_submission_batch 的入参不兼容；
2. get_workflow_state 的恢复身份不兼容。

因此，当前完整 Live E2E 不能宣告 PASS。排序后的导出和 Provider 跨会话恢复应停在 integration_required，而不是用 Mock、旧 ID 或字段猜测把流程接过去。

详细差异见 [远程 MCP 工具运行时审计](MCP_TOOL_RUNTIME_AUDIT_2026-07-23.md)。

## 先选对测试层

| 你改了什么 | 先跑什么 | 它能证明什么 | 它不能证明什么 |
| --- | --- | --- | --- |
| docs/ 中的人类入口文档 | npm run verify:docs | 生成区块仍与 Spec/变更记录同步 | 叙述是否反映远端真实写入。 |
| Hook、Plugin、Skill reference | npm run test:fast | TypeScript 构建和 Node Hook 回归通过；包括外发确认的跨 turn 一次性回执 | Provider、MySQL、企微真实可用。 |
| manifest、Skill 装载、OpenClaw 兼容性 | npm run test:openclaw | 源码可在隔离 OpenClaw 配置中加载，4 个 Hook 可被识别 | 桌面 UI 和真实业务写入。 |
| 多个本地面一起变更 | npm run test:headless | 依次跑 Fast + OpenClaw 装载检查 | 远端 MCP 的写路径。 |
| 合并或发布候选 | npm run verify | 仓库治理、测试、扫描和打包前结构检查 | 外发、回填、导出的真实业务结果。 |
| 远端 Tool Schema | npm run verify:provider | 只读比较远端 tools/list 与本地契约 | 成功出参和写行为。 |

npm run verify:provider 目前**预期返回非零**：这是已知远端 Schema 漂移的报警，不是可以忽略的绿色测试。它只执行初始化和 tools/list，不调用业务写 Tool。

## 最短开发循环

~~~bash
# 文档改动
npm run verify:docs
git diff --check

# Hook / Plugin / Tool reference 改动
npm run test:fast
git diff --check

# 还改了 Skill、manifest 或 OpenClaw 装载面
npm run test:headless
git diff --check
~~~

在提交或发布候选前，再跑：

~~~bash
npm run verify
npm run pack:yp
~~~

不要为了让一次快速测试更快而跳过最贴近改动的测试；也不要每改一行文案就跑一次真实外发。

## 真正联调前的准备

真实联调需要一个隔离、可回收环境。开始前逐项确认：

- 使用刚生成且版本唯一的安装包，而不是源码目录或同名旧 tgz；
- MCP 指向测试/开发环境，测试机构和企微群与生产隔离；
- Brief 脱敏，但仍包含平台、数量、截止和预算档位；
- 记录项目命名前缀、测试机构、清理负责人和允许的副作用；
- 不把客户原文、凭据、完整响应或企微正文写入仓库。

任何一项缺失，结论写 BLOCKED，不要改用生产环境或 Mock 冒充真实结果。

## 当前应怎样跑 Live E2E

先把它当作“找第一个真实阻断点”，而不是必须跑到最后的表演。每一跳都保存脱敏证据，遇到未知写入或契约不兼容立即停止。

~~~text
脱敏 Brief
→ validate_requirement
→ search_creators
→ rank_mcns
→ 用户在网页中选择询价字段
→ create_with_distributions（Hook 先弹精确企微外发确认）
→ 用户明确确认后，一次性调用 Provider
→ 只对明确 sent 的机构同步
→ 回填 / ingest（Provider 能力满足时）
→ 手工拓展（满足入口条件时）
→ rank_creators
→ integration_required（当前导出契约阻断）
~~~

如果用户一开始明确要手工拓展，且尚未启动搜索，则可走较短路径：

~~~text
validate_requirement
→ manual_source_creators(requirement_id, size)
→ 确认返回非空达人列表
→ rank_creators
→ integration_required（当前批次导出阻断）
~~~

manual_source_creators 的 size 是字符串，例如 "10"；requirement_id 必须是刚刚校验得到的 32 位 data.id。更完整的规则在 [手工拓展达人：当前流程与边界](MANUAL_SOURCING_WORKFLOW_OPTIMIZATION_PLAN.md)。

## 外发和结果未知的处理

当前 `create_with_distributions` 不能跳过本地确认。每次要实际外发时，Hook 会先阻断 Provider 调用并返回精确的 `AskUserQuestion`；宿主应原样展示其中的机构、字段和消息正文。用户明确选择“确认发送”后，最新未过期回执可跨用户回调的 turn 放行下一次调用一次，随后消费。当前实现不会重新比对下一次调用参数与弹窗参数是否相等，因此它只能授权这一次动作，不能被当成长期或可重复的授权。

取消、拒绝、关闭、超时或回调失败时，没有 Provider 外发；无 host session 时，插件仅使用自己的全局 fallback 回执。批量发送后若需要逐机构 fallback，每一家都必须走自己的确认。正确验收仍不是“调用没有抛错”，而是逐机构检查：

| 观察到的结果 | 能否说已发送 | 下一步 |
| --- | --- | --- |
| 每个机构都有可关联的明确 sent 状态 | 可以，仅针对这些机构 | 用同一项目和这些机构调用同步 Tool。 |
| 只有顶层 success、汇总名单或请求回显 | 不可以 | 保留原响应，等待可验证的逐机构证据。 |
| 明确 failed / unbound | 不可以 | 按实际失败原因处理；不要重发已 sent 的机构。 |
| 超时、连接断开、响应丢失 | 不可以，状态未知 | 停止并对账；禁止盲目再调外发 Tool。 |
| 确认弹窗被取消、拒绝、关闭或超时 | 不适用：Provider 不应被调用 | 等待新的用户指令；如需再次外发，重新走确认。 |

同步成功也不能反过来证明企微已发送。外发证据只来自 create_with_distributions 的实际逐机构回执。

## 记录什么，才算可复现

一次 Live E2E 至少记录下列脱敏信息：

| 检查点 | 要保存的证据 |
| --- | --- |
| 需求校验 | 本次返回的 data.id、业务错误码或 trace。 |
| 搜索与 MCN 排序 | 实际供给摘要、选中机构集合和 inquiry 关联证据。 |
| 网页字段选择 | 用户实际提交的字段结果，不保存敏感正文。 |
| 外发 | 精确确认弹窗与明确选择、每个机构的实际发送状态和关联项目 ID。 |
| 手工拓展 | 请求的 size、非空达人数组是否收到及已展示标记。 |
| 排序 | 当前 requirement 的实际排序结果或错误。 |
| 阻断 | 首个失败 Tool、参数形状、错误码、trace、预期行为。 |

修复后应从新的测试项目和新的会话重跑相关链路。不要把旧会话中的 ID、旧响应或人工补写的状态当作新版本证据。

## 常见结果如何写

只使用以下结论：

- PASS：该范围内的所有必要检查点都已实际执行并通过；
- FAIL：已执行，且看到确定错误；
- BLOCKED：缺少环境、权限、契约或安全对账条件，不能继续；
- NOT RUN：根本未执行。

例如，npm run test:fast 通过但 create_submission_batch 被运行时阻断，应写“本地回归 PASS，完整 Live E2E BLOCKED”，不能写“系统整体通过”。
