# YPmcn 高效联调测试指南

## 1. 目标

日常开发不再从“打包插件 → 安装到 YP Action → 输入需求 → 人工观察断点”开始。测试按成本由低到高分为 Fast、OpenClaw Headless、完整验证和 YP Action UI 四层；前一层失败时直接修复，不进入下一层。

```text
代码修改
  → Fast：契约、Hook、状态轨迹、MCP 协议
  → OpenClaw Headless：真实 YP OpenClaw 装载
  → Verify：全仓库发布前验证
  → YP Action UI：安装器与桌面端冒烟
```

## 2. 环境准备

在 Git 根目录执行命令：

```bash
cd <repo-root>
npm ci
```

要求：

- 使用根 `package-lock.json`，不要分别安装 workspace 依赖。
- Python 命令使用 `uv`，禁止 `pip`。
- `npm run test:openclaw` 默认使用：

```text
/Applications/YP Action.app/Contents/Resources/cfmind/openclaw.mjs
```

如需检查另一份 OpenClaw CLI，可显式覆盖：

```bash
OPENCLAW_CLI=/absolute/path/to/openclaw.mjs npm run test:openclaw
```

测试使用临时 OpenClaw 配置和临时 Hook 状态文件，不读取或修改 YP Action 正式用户配置。当前 YP Action `2026.7.1` 内置 OpenClaw `2026.4.14`，仓库开发依赖与 Headless 基线同步到该版本。

源码 MCP 配置默认使用受控开发 profile；切换命令会同步更新 `spec/mcp.json` 的 active endpoint、`YPmcn/mcp.json` 与 `YPmcn/.mcp.json`：

```bash
npm run mcp:dev   # http://192.168.0.129:32008/sse
npm run mcp:prod  # https://mcp.eshypdata.com/sse
```

发布暂存始终强制写入生产 SSE，不会把局域网开发地址打进安装包。`YPMCN_MCP_URL` 只用于 Headless 临时覆盖，不写正式配置。

### 2.1 远程 MCP 联调地址

联调使用远程开发机 `32008` 端口的 SSE endpoint：

```text
局域网：http://192.168.0.129:32008/sse
Tailscale：http://100.82.209.65:32008/sse
```

已确认该 endpoint 的 Server 为 `YP Local Business MCP` `1.9.4`，支持 MCP 协议 `2024-11-05`。优先使用局域网地址；不在同一局域网时使用 Tailscale 地址。

可用业务工具面为：

```text
business_health
select_inquiry_form_fields
validate_requirement
search_creators
rank_mcns
rank_creators
create_submission_batch
create_with_distributions
ingest_mcn_submissions
sync_mcn_inquiry_status
manual_source_creators
record_client_feedback
audit_manual_adjustment
get_creator_detail
get_recommendation_run_detail
get_workflow_state
```

`tools/list` 仍可能广告 `pgy_` 前缀的工具，但联调中必须全部禁用：客户端不展示、Agent 不选择、测试不调用，也不得将其结果作为联调证据。当前禁用项包括：

```text
pgy_kol_quote_turbo
pgy_kol_quote_excel
pgy_kol_quote_detail
pgy_notes_publish_data
pgy_kol_info_turbo
```

禁用规则按前缀 `pgy_` 判定，不仅限于上述当前清单；服务端后续新增的任何 `pgy_` 工具也自动属于禁用面。

## 3. 如何选择测试命令

| 修改类型 | 必跑命令 | 目的 |
|---|---|---|
| Python Hook、状态机、内部实现 | `npm run test:fast` | 验证完整轨迹和 MCP 协议 |
| Skill、Plugin、manifest、OpenClaw 适配 | `npm run test:openclaw` | 验证 YP 同源运行时能发现并装载 |
| MCP stdio server | `npm run test:mcp` | 快速验证 JSON-RPC 协议面 |
| 合并前 | `npm run test:headless` | 串联 Fast 与 OpenClaw Headless |
| 高风险或发布候选 | `npm run verify` | 全仓库验证 |
| 发布安装包 | `npm run pack:yp` | 构建并扫描 tgz |
| 开发 MCP 契约只读检查 | `npm run verify:provider` | 独立确认开发机 tools/list，不调用写 Tool |
| 生产 MCP 路由诊断 | `npm run verify:provider:prod` | 只读确认生产 SSE；生产域名恢复前预期失败 |

### 3.1 日常默认：Fast

```bash
npm run test:fast
```

该命令覆盖：

1. 构建并测试 `YPmcn` Plugin。
2. 验证 Native Plugin 到 Python Hook 的实际桥接。
3. 回放完整业务状态轨迹及非法越级、确认门禁、语义 ID、终态锁。
4. 构建 vector MCP。
5. 启动真实 MCP 子进程，发送 `initialize`、`notifications/initialized` 和 `tools/list`。
6. 检查工具列表与 `inputSchema` 的基本协议完整性。

成功标志至少包括：

```text
18 tests passed
25 PASS, 0 FAIL
[smoke:mcp] PASS protocol=2024-11-05 tools=3
```

具体测试数量可能随功能增加而上升，不应通过删除或放宽测试维持固定数字。

### 3.2 修改 MCP 时：协议 Smoke

```bash
npm run test:mcp
```

它直接启动 `vector-mcp/dist/server.js`，能发现内部函数测试覆盖不到的问题：

- server 无法启动；
- stdout 混入非 JSON 日志；
- MCP initialize 响应错误；
- `tools/list` 为空或 schema 异常；
- 构建产物、环境变量或工作目录错误。

协议日志必须写到 stderr。任何写入 stdout 的调试输出都可能破坏 stdio MCP。

### 3.3 修改 Plugin/Skill 时：OpenClaw Headless

```bash
npm run test:openclaw
```

该命令使用 YP Action 内置 OpenClaw，但不启动 Electron UI。它会：

1. 创建临时 `OPENCLAW_STATE_DIR` 和配置文件；
2. 从 `YPmcn/` 源码目录装载 Plugin；
3. 从 `YPmcn/skills/` 发现 Skill；
4. 检查 Plugin 成功 imported/loaded 且 diagnostics 为空；
5. 检查 `媒介助手` Skill eligible；
6. 检查 `before_tool_call`、`after_tool_call`、`session_end` 三个 Hook 被真实运行时识别；
7. 删除临时状态目录。

成功标志：

```text
[smoke:openclaw] PASS plugin=loaded skill=媒介助手
```

这层主要发现：

- manifest、package entry 或构建产物错误；
- OpenClaw API/Hook 名称不兼容；
- Skill frontmatter 或目录结构错误；
- 源码能编译，但 YP 内置 OpenClaw 无法装载。

### 3.4 合并前：Headless 闭环

```bash
npm run test:headless
git diff --check
git status --short
```

`test:headless` 依次执行 `test:fast` 和 `test:openclaw`。测试结束后检查工作区，正式状态文件不应因测试发生变化：

```bash
git diff -- YPmcn/state/session_guard.json
```

预期无输出。如果出现差异，优先修复测试隔离，不要提交测试生成的会话状态。

## 4. 用一个需求定位 Agent 卡点

`test:fast` 和 `test:openclaw` 不会运行模型。它们分别证明 Hook/协议正确和 Plugin/Skill 能被装载，**不能证明 Agent 收到需求后会选对 Skill、调用正确 Tool 或继续下一步**。定位 Agent 卡点必须单独跑一次需求驱动测试。

### 4.1 测试前先定义终点

一次只测试到一个安全终点，不要直接要求 Agent 完成全链路。推荐先测“需求校验完成并准备搜索达人”，成功标准为：

```text
用户需求被识别
→ 媒介助手被采用
→ 调用 validate_requirement
→ 得到 success=true、data.id 和 data.status=ready
→ Agent 下一步准备调用 search_creators
```

如果本轮只验证路由，可把终点设为“Agent 明确准备调用 `validate_requirement`”，不要允许生产写调用。涉及外发、建联、提交批次等写操作时，必须使用专用测试环境或停在人工确认门禁前。

### 4.2 最小虚构需求

在 YP Action 新会话输入以下需求，避免旧上下文影响判断：

```text
帮我找 10 位小红书美妆达人，粉丝 10 万到 50 万，单人预算 5000 到 10000 元，
档期 2026 年 9 月，目标是新品种草。先完成需求校验并告诉我下一步，不要建联、外发或创建提交批次。
```

年份、平台、数量和预算口径必须明确，否则 Agent 停下来追问可能是正确行为，不应直接判为卡死。

也可以通过 YP Action 当前 Gateway 运行同一轮并保留 JSON 结果：

```bash
OPENCLAW_CLI="/Applications/YP Action.app/Contents/Resources/cfmind/openclaw.mjs"
SESSION_ID="ypmcn-debug-$(date +%Y%m%d-%H%M%S)"

"$OPENCLAW_CLI" agent \
  --session-id "$SESSION_ID" \
  --message '帮我找 10 位小红书美妆达人，粉丝 10 万到 50 万，单人预算 5000 到 10000 元，档期 2026 年 9 月，目标是新品种草。先完成需求校验并告诉我下一步，不要建联、外发或创建提交批次。' \
  --verbose on \
  --json
```

这条命令使用当前 Gateway、模型和已安装配置，不具备测试隔离。不要加 `--deliver`；如果当前配置连接生产 MCP，仍须按生产调用对待。

### 4.3 按检查点定位第一处断点

只记录**第一个未通过检查点**，先修这一层再重跑同一需求：

| 检查点 | 通过证据 | 未通过时检查 |
|---|---|---|
| 1. 会话进入 Agent | 返回本轮 JSON，且 session ID 一致 | Gateway、模型凭据、超时和会话路由 |
| 2. Skill 路由 | 回复/轨迹体现媒介助手规则和工具链 | Skill eligible、描述、触发语和上下文污染 |
| 3. 需求解析 | 平台、数量、预算口径、档期未被曲解 | `requirement-intake.md`、Tool 描述；合理追问不算失败 |
| 4. Tool 选择 | 首个业务 Tool 为 `validate_requirement` | Skill 指令、可见 Tool 列表、模型选 Tool |
| 5. Hook 放行 | 没有错误的 deny；或 deny 有稳定错误码 | 当前 phase、确认门禁、语义 ID、Hook 输入 |
| 6. MCP 调用 | 有请求、响应或明确的连接错误 | MCP 配置、连接、schema、认证和超时 |
| 7. 结果识别 | 识别 `success`、`data.id`、`data.status` | provider 返回结构和 after-tool 成功证据 |
| 8. 状态推进 | 会话状态记录 requirement ID 和下一 phase | `session_guard.json` 隔离副本、after-tool Hook |
| 9. Agent 续跑 | status=ready 后准备搜索；非 ready 时准确报告缺项 | Skill 停止条件、上下文长度、模型提前结束 |

“Agent 没继续”不是一个可执行结论。必须写成例如：

```text
FAIL at 4 Tool 选择
evidence: Agent 直接回答达人名单，未调用 validate_requirement
expected: 首个业务 Tool 为 validate_requirement
session: ypmcn-debug-20260716-153000
```

### 4.4 最小诊断记录

每轮只保留以下信息，客户原文和完整生产响应必须脱敏：

```text
需求版本：
会话 ID：
测试终点：
最后通过检查点：
第一个失败检查点：
Agent 最后一条可见行为：
Tool 名称 / Hook 错误码 / MCP 错误：
期望行为：
结论：PASS / FAIL / BLOCKED
```

修复后必须使用同一条需求和一个全新的 session ID 复测。只有原失败检查点通过且没有引入更早断点，才算修复有效。

## 5. 完整业务轨迹如何验证

当前确定性轨迹位于 `tests/test_hooks.py`，不调用模型、不消耗额度、不访问生产写接口。主链包括：

```text
validate_requirement
→ search_creators
→ rank_mcns
→ select_inquiry_form_fields
→ create_with_distributions
→ sync_mcn_inquiry_status
→ ingest_mcn_submissions
→ sync_mcn_inquiry_status
→ rank_creators
→ create_submission_batch
→ record_client_feedback
```

新增业务阶段或门禁时，至少补充：

- 正常路径的 pre-tool allow；
- 非法阶段的 deny 与稳定错误码；
- 成功结果触发的状态推进；
- 缺少成功证据时保持原阶段；
- 关键语义 ID 的提取与一致性检查；
- 重复写、终态、人工确认与恢复路径。

测试数据必须使用虚构 ID 和脱敏 payload，不得写入真实客户 Brief、凭据或完整生产响应。

## 6. 失败定位顺序

### Fast 构建失败

先单独运行组件测试：

```bash
npm --workspace YPmcn test
npm --workspace vector-mcp test
```

优先处理第一个 TypeScript 或测试错误，不要先打包重试。

### Python Hook 失败

单独运行：

```bash
uv run --no-project python -B tests/test_hooks.py
```

按输出定位：

- `BLOCKED_PHASE_MISMATCH`：阶段与允许工具矩阵不一致；
- `BLOCKED_CONFIRMATION_REQUIRED`：外发确认条件不完整；
- `BLOCKED_SEMANTIC_ID_MISMATCH`：输入 ID 与会话证据不一致；
- `WRITE_RESULT_UNKNOWN`：MCP 成功结果缺少状态推进所需证据；
- `INTEGRATION_REQUIRED`：运行时或 provider 契约不足，禁止模拟成功。

### MCP Smoke 失败

```bash
npm run test:mcp
```

- `Non-JSON MCP stdout`：检查 server 是否向 stdout 打印了日志；
- `Timed out waiting for initialize/tools/list`：检查进程启动、异步阻塞和 stdin readline；
- `tools/list returned no tools`：检查 Tool 注册和构建产物；
- server 提前退出：查看 smoke 输出附带的 stderr。

### OpenClaw Headless 失败

```bash
npm run test:openclaw
```

- `plugin ... not loaded`：检查 `openclaw.plugin.json`、`package.json` 和 `dist/index.js`；
- diagnostics 非空：以 OpenClaw 返回的首个诊断为准；
- Skill not eligible：检查 `SKILL.md` frontmatter、目录和运行要求；
- Hook 数量或名称不符：检查 `api.on(...)` 注册名，禁止使用运行时不支持的 Hook；
- CLI 路径不存在：确认 YP Action 已安装，或用 `OPENCLAW_CLI` 指向待测版本。

## 7. 何时运行完整 Verify

日常小改不必反复运行全量验证。以下情况运行一次：

- 合并或发布前；
- 修改 Spec、manifest、发布包内容；
- 修改根安装图、依赖或 workspace；
- 修改安全扫描、provider comparator 或文档同步；
- 相邻测试不足以覆盖风险；
- Headless 出现跨组件失败。

```bash
npm run verify
```

`verify` 包含 Spec、文档同步、安装图、密钥扫描、Plugin、Python Hook、provider comparator、Skill 文档、vector MCP 和发布包结构验证。开发 Provider 独立证据由默认只读命令提供；生产路由仅作诊断：

```bash
npm run verify:provider
npm run verify:provider:prod
```

开发机绿色不得描述为生产 provider 已通过。

## 8. YP Action 发布候选冒烟

只有 Headless 和完整验证通过后才打包：

```bash
npm run pack:yp
```

使用 `packages/releases/` 下新生成的 tgz 安装。YP Action UI 只验证 Headless 无法覆盖的桌面集成面：

1. 安装器能识别正确 Plugin ID 和版本；
2. Plugin、Skill、MCP 配置同步成功；
3. 新会话能发现“媒介助手”；
4. 输入最小虚构需求后，模型选择正确 MCP Tool；
5. 未确认外发时 Native Hook 能阻断；
6. 正常 Tool 结果能推进会话状态；
7. 重启 Gateway/YP Action 后启用状态和配置仍存在；
8. Gateway 日志没有 unknown hook、plugin load error 或 MCP connection error。

UI 冒烟失败时，先归类：

- Headless 也失败：回到源码和运行时装载层修复；
- 只有安装后失败：检查 tgz 内容、安装器和配置同步；
- 只有模型选 Tool 错误：检查 Skill 路由、Tool 描述和会话上下文；
- 只有桌面交互失败：检查 YP Action callback/UI 层；
- 只有生产调用失败：运行只读 provider 检查，禁止用 Mock 结果替代。

## 9. 安全与清洁要求

- Headless 测试不得复用或修改 `~/Library/Application Support/YP Action` 下的正式状态。
- `verify:provider` 与 `verify:provider:prod` 只发送 `initialize`、`notifications/initialized`、`tools/list`；自动测试不得调用生产业务 Tool。
- 自动测试不得调用外发、写数据库或不可逆 Tool。
- 测试状态必须写入临时目录，结束后自动删除。
- 不把日志、客户 Brief、完整 payload、凭据或未脱敏响应写入仓库。
- 不直接编辑 `dist/`、`packages/.staging/` 或 tgz。
- 失败测试必须报告 FAIL；未运行必须报告 NOT RUN。

## 10. 最短开发循环

常规修改：

```bash
npm run test:fast
git diff --check
```

Plugin、Skill 或 OpenClaw 修改：

```bash
npm run test:headless
git diff --check
```

发布候选：

```bash
npm run test:headless
npm run verify
npm run pack:yp
```

最后只在 YP Action 中执行一次发布冒烟，不再把桌面 UI 当作首轮调试器。
