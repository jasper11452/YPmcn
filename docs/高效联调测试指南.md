# YPmcn 高效联调测试指南

## 1. 目标

日常开发先跑低成本测试，但涉及 Agent 行为、MCP、数据库和企微的版本，最终必须增加一条独立真实全链路。单元脚本不能替代 Agent E2E。

```text
代码修改
  → Fast：契约、Hook、状态轨迹、MCP 协议
  → OpenClaw Headless：真实 YP OpenClaw 装载
  → Verify：全仓库发布前验证
  → YP Action UI：安装器与桌面端冒烟
  → Live E2E：真实 Brief、Agent、远程 MCP、开发库、测试企微、跨 session 恢复与 CSV
```

## 2. 环境准备

在 Git 根目录执行命令：

```bash
cd <repo-root>
npm ci
```

要求：

- 使用根 `package-lock.json`，不要分别安装 workspace 依赖。
- Python 命令使用 `uv`，禁止 `pip`。
- `npm run test:openclaw` 默认使用：

```text
/Applications/YP Action.app/Contents/Resources/cfmind/openclaw.mjs
```

如需检查另一份 OpenClaw CLI，可显式覆盖：

```bash
OPENCLAW_CLI=/absolute/path/to/openclaw.mjs npm run test:openclaw
```

测试使用临时 OpenClaw 配置和临时 Hook 状态文件，不读取或修改 YP Action 正式用户配置，也不要求远程 MCP 在线；远程 `tools/list` 由 `verify:provider` 单独检查。当前 YP Action `2026.7.1` 内置 OpenClaw `2026.4.14`，仓库开发依赖与 Headless 基线同步到该版本。

源码 MCP 配置默认使用受控开发 profile；切换命令会同步更新 `spec/mcp.json` 的 active endpoint、`YPmcn/mcp.json` 与 `YPmcn/.mcp.json`：

```bash
npm run mcp:dev   # https://mcp.eshypdata.com/sse
npm run mcp:prod  # https://mcp.eshypdata.com/sse
```

开发与生产 profile 统一连接 `https://mcp.eshypdata.com/sse`。安装包只包含该远程业务 MCP 配置，不再包含 `vector-mcp`。YP Action/OpenClaw **支持 SSE**，当前 Headless 也能通过 `openclaw mcp set` 注册 `transport: "sse"`；“OpenClaw 只支持 stdio”的旧判断已经失效。

尚未被自动测试证明的是安装配置格式：包内 `.mcp.json` 使用 `{ "mcpServers": { ... } }`，由 Codex/Claude 风格的 bundle manifest 引用；OpenClaw CLI 则把其中单个 server object 通过 `mcp set <name> <server-json>` 写入自己的 MCP 注册表。`test:openclaw` 当前显式完成了这一步转换，所以它能证明 SSE 可注册，不能单独证明 YP Action 安装器会在全新用户环境自动导入包内文件。只有在没有预存 `ypmcn-mcp` 全局配置的干净 YP Action 环境安装 tgz 后，`openclaw mcp list` 自动出现该 entry 且能发现业务 Tool，才可写“安装插件即安装 MCP”。

### 2.1 远程 MCP 联调地址

开发、联调、打包与生产统一使用：

```text
https://mcp.eshypdata.com/sse
```

不再使用 Tailscale、开发机局域网地址、SSH 隧道或本地端口转发。只读契约检查直接运行 `npm run verify:provider`；网络或服务不可达时报告真实失败，不得改用本地地址或 Mock 冒充远程证据。

可用业务工具面为：

```text
business_health
select_inquiry_form_fields
validate_requirement
search_creators
rank_mcns
rank_creators
create_submission_batch
create_with_distributions
ingest_mcn_submissions
sync_mcn_inquiry_status
manual_source_creators
record_client_feedback
audit_manual_adjustment
get_creator_detail
get_recommendation_run_detail
get_workflow_state
```

`tools/list` 仍可能广告 `pgy_` 前缀的工具，但联调中必须全部禁用：客户端不展示、Agent 不选择、测试不调用，也不得将其结果作为联调证据。当前禁用项包括：

```text
pgy_kol_quote_turbo
pgy_kol_quote_excel
pgy_kol_quote_detail
pgy_notes_publish_data
pgy_kol_info_turbo
```

禁用规则按前缀 `pgy_` 判定，不仅限于上述当前清单；服务端后续新增的任何 `pgy_` 工具也自动属于禁用面。

## 3. 如何选择测试命令

| 修改类型 | 必跑命令 | 目的 |
|---|---|---|
| Native Node Hook、契约、内部实现 | `npm run test:fast` | 验证当前安全门禁和 MCP 协议 |
| Skill、Plugin、manifest、OpenClaw 适配 | `npm run test:openclaw` | 验证 YP 同源运行时能发现并装载 |
| MCP stdio server | `npm run test:mcp` | 快速验证 JSON-RPC 协议面 |
| 合并前 | `npm run test:headless` | 串联 Fast 与 OpenClaw Headless |
| 高风险或发布候选 | `npm run verify` | 全仓库验证 |
| 发布安装包 | `npm run pack:yp` | 构建并扫描 tgz |
| 开发 MCP 契约只读检查 | `npm run verify:provider` | 独立确认开发机 tools/list，不调用写 Tool |
| 生产 MCP 路由诊断 | `npm run verify:provider:prod` | 只读确认生产 SSE；生产域名恢复前预期失败 |
| 业务发布候选 | 本文 §4 Live E2E | 真实 Agent、MCP、开发库、测试企微与跨 session 恢复 |

### 3.1 日常默认：Fast

```bash
npm run test:fast
```

该命令覆盖：

1. 构建并测试 `YPmcn` Plugin 及 Native Node Hook。
2. 验证 Brief、供给方案、外发确认和 unknown outcome 的 fail-closed 门禁。
3. 构建 vector MCP。
4. 启动真实 MCP 子进程，发送 `initialize`、`notifications/initialized` 和 `tools/list`。
5. 检查工具列表与 `inputSchema` 的基本协议完整性。

成功标志至少包括：

```text
[YPmcn tests] pass
[smoke:mcp] PASS protocol=2024-11-05 tools=3
```

具体测试数量可能随功能增加而上升，不应通过删除或放宽测试维持固定数字。

### 3.2 修改 MCP 时：协议 Smoke

```bash
npm run test:mcp
```

它直接启动 `vector-mcp/dist/server.js`，能发现内部函数测试覆盖不到的问题：

- server 无法启动；
- stdout 混入非 JSON 日志；
- MCP initialize 响应错误；
- `tools/list` 为空或 schema 异常；
- 构建产物、环境变量或工作目录错误。

协议日志必须写到 stderr。任何写入 stdout 的调试输出都可能破坏 stdio MCP。

### 3.3 修改 Plugin/Skill 时：OpenClaw Headless

```bash
npm run test:openclaw
```

该命令使用 YP Action 内置 OpenClaw，但不启动 Electron UI。它会：

1. 创建临时 `OPENCLAW_STATE_DIR` 和配置文件；
2. 从 `YPmcn/` 源码目录装载 Plugin；
3. 从 `YPmcn/skills/` 发现 Skill；
4. 检查 Plugin 成功 imported/loaded 且 diagnostics 为空；
5. 检查 `媒介助手` Skill eligible；
6. 检查 `before_prompt_build`、`before_tool_call`、`after_tool_call`、`session_end` 四个 Hook 被真实运行时识别；
7. 删除临时状态目录。

成功标志：

```text
[smoke:openclaw] PASS plugin=loaded skill=媒介助手
```

这层主要发现：

- manifest、package entry 或构建产物错误；
- OpenClaw API/Hook 名称不兼容；
- Skill frontmatter 或目录结构错误；
- 源码能编译，但 YP 内置 OpenClaw 无法装载。

### 3.4 合并前：Headless 闭环

```bash
npm run test:headless
git diff --check
git status --short
```

`test:headless` 依次执行 `test:fast` 和 `test:openclaw`。测试结束后检查工作区，正式状态文件不应因测试发生变化：

```bash
git diff -- YPmcn/state/session_guard.json
```

预期无输出。如果出现差异，优先修复测试隔离，不要提交测试生成的会话状态。

## 4. 独立真实 Live E2E

`test:fast` 和 `test:openclaw` 不运行模型，也不访问真实业务链。每个准备交付的业务版本必须由一个未参与实现的 verifier，用一条脱敏真实 Brief 跑完整链路；不能再用“本地脚本模拟了每个 Tool”代替。

当前执行状态（2026-07-18）：`BLOCKED / NOT RUN`。本次只建立了独立真实验收流程，没有产生 Live E2E 通过证据；开发 Provider 的 inquiry sync/ledger 断点尚未完成远程验证。开发与生产现统一使用公开 SSE endpoint；只读契约检查仍不等于真实业务链通过，在完整链路跑完前不得标记 `PASS`。

### 4.1 前置条件

- YP Action 使用待验插件，MCP 指向 `https://mcp.eshypdata.com/sse`。
- 开发 MySQL、Qdrant、embedding/rerank 和外部 API 都是测试环境。
- 只选择专用测试机构/企微群，项目名前缀固定为 `E2E-YPmcn-<日期>-<唯一标记>`。
- Brief 必须来自真实需求结构但已脱敏，明确平台、数量、提报截止和单达人预算档位。
- 媒介本人在 UI 中完成供给方案、机构/字段/消息和不可逆外发确认；自动化不得代点。

任一前置条件不满足，结果写 `BLOCKED`，不得切到生产或用 Mock 冒充。

### 4.2 唯一验收链

```text
真实 Brief
→ Agent 三遍解析；缺失/歧义时 AskUserQuestion
→ validate_requirement
→ search_creators（真实 SQL/Qdrant/rerank provenance）
→ 展示十项供给方案与同量纲机构覆盖/手扒账号比例 → Ask 确认
→ rank_mcns → 媒介选机构
→ select_inquiry_form_fields → 固定企微消息预览 → Ask 确认
→ create_with_distributions（测试企微）
→ sync_mcn_inquiry_status
→ 用测试表单提交至少一条真实回填
→ sync → ingest_mcn_submissions → sync
→ rank_creators
→ create_submission_batch
→ 固定列 export_csv
→ 新建 session，仅用 demand_id + demand_version 调 get_workflow_state 恢复
```

必须跑到 CSV 与新 session 恢复；只跑到 `validate_requirement` 或本地 Hook 回放不算 Live E2E。

### 4.3 检查点与证据

| 检查点 | 必须看到的真实证据 |
|---|---|
| Brief 解析 | 单达人预算/档位、截止、数量准确；每条原文条件有字段、原文保留或澄清去向 |
| Agent/Hook | 首个业务 Tool 正确；未确认时稳定阻断；确认后只放行同一指纹 |
| Search | MCP 返回真实候选、`retrieval_mode`、provenance、去重硬筛数和完整 `supply_plan` |
| 外发 | 外部 project/distribution ID、测试企微消息、表单链接与选中机构一致 |
| 回收 | sync 读取真实 distribution；同一 inquiry 被更新而非重复创建 |
| 精排 | 只使用已回收/有效手扒候选；硬条件违规为 0；run_id 可查 |
| CSV | 固定 12 列、顺序和文件名正确，空值不伪造成 0 |
| 恢复 | 全新 session 恢复同一 phase、ID 和唯一下一步，不依赖旧聊天上下文 |
| 幂等 | 对同一测试写请求做受控重复调用，资源 ID 不变；unknown 不自动重发 |

只记录第一个失败检查点，并保存脱敏 trace/session、Tool 名、Hook 错误码、MCP 错误、相关业务 ID 和期望行为。修复后使用同一 Brief、全新 session 与新的测试项目标记重跑整条链，不能从断点后直接续测。

### 4.4 通过标准

结论只能是 `PASS`、`FAIL` 或 `BLOCKED`。全部检查点通过且 verifier 与实现者不是同一推理链才可写 PASS；任何真实外部环节未运行均不得写“基本通过”。

## 5. 确定性 Hook 与业务链如何验证

当前执行面的确定性门禁位于 `YPmcn/tests/native-hooks.test.mjs`：它验证契约、Brief 必填、Ask 指纹绑定、外发确认和 unknown 后禁止盲重试。Node Hook 不推进数据库 phase，因此本地测试不得模拟或声称完整业务恢复链已通。

`tests/test_hooks.py` 及 `YPmcn/hooks/*.py` 仅是历史回归工件，已从 `test:fast` 和 `verify` 移除；需要考古旧行为时可显式运行 `npm run test:legacy-hooks`，其旧 phase 不是当前 workflow 证据。

真实业务链只能按 §4 的 Live E2E 验收；本地 Hook 回归不能替代 MCP、开发库、测试企微、回收和跨 session 恢复证据。

## 6. 失败定位顺序

### Fast 构建失败

先单独运行组件测试：

```bash
npm --workspace YPmcn test
npm --workspace vector-mcp test
```

优先处理第一个 TypeScript 或测试错误，不要先打包重试。

### Native Node Hook 失败

单独运行：

```bash
npm --workspace YPmcn test
```

按输出定位：

- `BLOCKED_REQUIREMENT_INCOMPLETE`：Brief 必填、格式或单达人预算档位不完整；
- `YP_SUPPLY_PLAN_CONFIRMATION_REQUIRED`：供给方案尚未用十个固定字段确认；
- `YP_CONFIRMATION_REQUIRED`：外发请求尚未绑定一次性 Ask 确认；
- `BLOCKED_UNKNOWN_OUTCOME`：外发结果未知，禁止盲重试；
- `INTEGRATION_REQUIRED`：运行时或 provider 契约不足，禁止模拟成功。

### MCP Smoke 失败

```bash
npm run test:mcp
```

- `Non-JSON MCP stdout`：检查 server 是否向 stdout 打印了日志；
- `Timed out waiting for initialize/tools/list`：检查进程启动、异步阻塞和 stdin readline；
- `tools/list returned no tools`：检查 Tool 注册和构建产物；
- server 提前退出：查看 smoke 输出附带的 stderr。

### OpenClaw Headless 失败

```bash
npm run test:openclaw
```

- `plugin ... not loaded`：检查 `openclaw.plugin.json`、`package.json` 和 `dist/index.js`；
- diagnostics 非空：以 OpenClaw 返回的首个诊断为准；
- Skill not eligible：检查 `SKILL.md` frontmatter、目录和运行要求；
- Hook 数量或名称不符：检查 `api.on(...)` 注册名，禁止使用运行时不支持的 Hook；
- CLI 路径不存在：确认 YP Action 已安装，或用 `OPENCLAW_CLI` 指向待测版本。

## 7. 何时运行完整 Verify

日常小改不必反复运行全量验证。以下情况运行一次：

- 合并或发布前；
- 修改 Spec、manifest、发布包内容；
- 修改根安装图、依赖或 workspace；
- 修改安全扫描、provider comparator 或文档同步；
- 相邻测试不足以覆盖风险；
- Headless 出现跨组件失败。

```bash
npm run verify
```

`verify` 包含 Spec、文档同步、安装图、密钥扫描、Plugin/Native Node Hook、provider comparator、Skill 文档、vector MCP 和发布包结构验证。开发 Provider 独立证据由默认只读命令提供；生产路由仅作诊断：

```bash
npm run verify:provider
npm run verify:provider:prod
```

开发机绿色不得描述为生产 provider 已通过。

## 8. YP Action 发布候选冒烟

只有 Headless 和完整验证通过后才打包：

```bash
npm run pack:yp
```

使用 `packages/releases/` 下新生成的 tgz 安装。YP Action UI 只验证 Headless 无法覆盖的桌面集成面：

1. 安装器能识别正确 Plugin ID 和版本；
2. Plugin、Skill 正常装载，包内只含远程 MCP 配置且不含 Vector MCP；在无预存 `ypmcn-mcp` 的干净测试状态中确认安装器是否把包内 `mcpServers` 文件转换并注册到 OpenClaw，不能用安装前已有的全局 entry 冒充自动安装；
3. 新会话能发现“媒介助手”；
4. 输入最小虚构需求后，模型选择正确 MCP Tool；
5. 未确认外发时 Native Hook 能阻断；
6. 正常 Tool 结果能推进会话状态；
7. 重启 Gateway/YP Action 后启用状态和配置仍存在；
8. Gateway 日志没有 unknown hook、plugin load error 或 MCP connection error。

UI 冒烟失败时，先归类：

- Headless 也失败：回到源码和运行时装载层修复；
- 只有安装后失败：检查 tgz 内容、安装器和配置同步；
- 只有模型选 Tool 错误：检查 Skill 路由、Tool 描述和会话上下文；
- 只有桌面交互失败：检查 YP Action callback/UI 层；
- 只有生产调用失败：运行只读 provider 检查，禁止用 Mock 结果替代。

## 9. 安全与清洁要求

- Headless 测试不得复用或修改 `~/Library/Application Support/YP Action` 下的正式状态。
- `verify:provider` 与 `verify:provider:prod` 只发送 `initialize`、`notifications/initialized`、`tools/list`；自动测试不得调用生产业务 Tool。
- CI/Headless 自动测试不得调用外发、写数据库或不可逆 Tool；§4 Live E2E 只在专用测试环境由媒介显式确认后执行。
- 测试状态必须写入临时目录，结束后自动删除。
- 不把日志、客户 Brief、完整 payload、凭据或未脱敏响应写入仓库。
- 不直接编辑 `dist/`、`packages/.staging/` 或 tgz。
- 失败测试必须报告 FAIL；未运行必须报告 NOT RUN。

## 10. 最短开发循环

常规修改：

```bash
npm run test:fast
git diff --check
```

Plugin、Skill 或 OpenClaw 修改：

```bash
npm run test:headless
git diff --check
```

发布候选：

```bash
npm run test:headless
npm run verify
npm run pack:yp
```

最后只在 YP Action 中执行一次发布冒烟，不再把桌面 UI 当作首轮调试器。
