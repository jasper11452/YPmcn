# YPmcn 高效联调测试指南

## 1. 目标

日常开发先跑低成本测试，但涉及 Agent 行为、MCP、数据库和企微的版本，最终必须增加一条独立真实全链路。单元脚本不能替代 Agent E2E。

```text
代码修改
  → Fast：契约、Hook、状态轨迹、MCP 协议
  → OpenClaw Headless：真实 YP OpenClaw 装载
  → Verify：全仓库发布前验证
  → YP Action UI：安装器与桌面端冒烟
  → Live E2E：真实 Brief、Agent、远程 MCP、开发库、测试企微、跨 session 恢复与 CSV
```

## 2. 环境准备

在 Git 根目录执行命令：

```bash
cd <repo-root>
npm ci
```

要求：

- 使用根 `package-lock.json`，不要分别安装 workspace 依赖。
- Python 命令使用 `uv`，禁止 `pip`。
- `npm run test:openclaw` 默认使用：

```text
/Applications/YP Action.app/Contents/Resources/cfmind/openclaw.mjs
```

如需检查另一份 OpenClaw CLI，可显式覆盖：

```bash
OPENCLAW_CLI=/absolute/path/to/openclaw.mjs npm run test:openclaw
```

测试使用临时 OpenClaw 配置和临时 Hook 状态文件，不读取或修改 YP Action 正式用户配置，也不要求远程 MCP 在线；远程 `tools/list` 由 `verify:provider` 单独检查。当前 YP Action `2026.7.1` 内置 OpenClaw `2026.4.14`，仓库开发依赖与 Headless 基线同步到该版本。

源码 MCP 配置默认使用受控开发 profile；切换命令会同步更新 `spec/mcp.json` 的 active endpoint、`YPmcn/mcp.json` 与 `YPmcn/.mcp.json`：

```bash
npm run mcp:dev   # https://mcp.eshypdata.com/sse
npm run mcp:prod  # https://mcp.eshypdata.com/sse
```

开发与生产 profile 统一连接 `https://mcp.eshypdata.com/sse`。安装包只包含该远程业务 MCP 配置，不再包含 `vector-mcp`。YP Action/OpenClaw **支持 SSE**，当前 Headless 也能通过 `openclaw mcp set` 注册 `transport: "sse"`；“OpenClaw 只支持 stdio”的旧判断已经失效。

2026-07-19 已在没有预存 MCP 的 YP Action `2026.7.1` 环境完成干净安装核对：安装器会暂存并复制原生 OpenClaw Plugin、记录 Plugin 状态并同步 Gateway 配置，但不会读取包内 `.mcp.json` 并写入 YP Action 的 MCP Store。该包同时包含原生 `openclaw.plugin.json`，OpenClaw 会按原生 Plugin 优先级装载，不能再把同一目录当作 Codex/Claude bundle 自动加载 `.mcp.json`。因此当前 tgz 的 MCP 文件内容正确且确实进入安装目录，但“安装插件即自动安装 MCP”在该宿主版本上不成立。

在 YP Action 增加正式的 Plugin MCP 导入、更新与卸载归属管理前，Live E2E 必须按 §4.1 显式创建 `ypmcn-mcp`。这只是受控宿主配置步骤，不得写成安装器自动导入已通过，也不得让 Plugin 通过安装脚本或直接改 SQLite 绕过宿主管理边界。

## 3. 如何选择测试命令

| 修改类型 | 必跑命令 | 目的 |
|---|---|---|
| Native Node Hook、契约、内部实现 | `npm run test:fast` | 验证当前安全门禁和 MCP 协议 |
| Skill、Plugin、manifest、OpenClaw 适配 | `npm run test:openclaw` | 验证 YP 同源运行时能发现并装载 |
| MCP stdio server | `npm run test:mcp` | 快速验证 JSON-RPC 协议面 |
| 合并前 | `npm run test:headless` | 串联 Fast 与 OpenClaw Headless |
| 高风险或发布候选 | `npm run verify` | 全仓库验证 |
| 发布安装包 | `npm run pack:yp` | 构建并扫描 tgz |
| 开发 MCP 契约只读检查 | `npm run verify:provider` | 独立确认开发机 tools/list，不调用写 Tool |
| 生产 MCP 路由诊断 | `npm run verify:provider:prod` | 只读确认生产 SSE；生产域名恢复前预期失败 |
| 业务发布候选 | 本文 §4 Live E2E | 真实 Agent、MCP、开发库、测试企微与跨 session 恢复 |

### 3.1 日常默认：Fast

```bash
npm run test:fast
```

该命令覆盖：

1. 构建并测试 `YPmcn` Plugin 及 Native Node Hook。
2. 验证 Brief preview 仅作提示、非外发 Tool 默认放行，以及企微外发的 AskUserQuestion 多行警告、弹窗指纹与后续调用参数指纹绑定。
3. 构建 vector MCP。
4. 启动真实 MCP 子进程，发送 `initialize`、`notifications/initialized` 和 `tools/list`。
5. 检查工具列表与 `inputSchema` 的基本协议完整性。

成功标志至少包括：

```text
[YPmcn tests] pass
[smoke:mcp] PASS protocol=2024-11-05 tools=3
```

具体测试数量可能随功能增加而上升，不应通过删除或放宽测试维持固定数字。

### 3.2 修改 MCP 时：协议 Smoke

```bash
npm run test:mcp
```

它直接启动 `vector-mcp/dist/server.js`，能发现内部函数测试覆盖不到的问题：

- server 无法启动；
- stdout 混入非 JSON 日志；
- MCP initialize 响应错误；
- `tools/list` 为空或 schema 异常；
- 构建产物、环境变量或工作目录错误。

协议日志必须写到 stderr。任何写入 stdout 的调试输出都可能破坏 stdio MCP。

### 3.3 修改 Plugin/Skill 时：OpenClaw Headless

```bash
npm run test:openclaw
```

该命令使用 YP Action 内置 OpenClaw，但不启动 Electron UI。它会：

1. 创建临时 `OPENCLAW_STATE_DIR` 和配置文件；
2. 从 `YPmcn/` 源码目录装载 Plugin；
3. 从 `YPmcn/skills/` 发现 Skill；
4. 检查 Plugin 成功 imported/loaded 且 diagnostics 为空；
5. 检查 `媒介助手` Skill eligible；
6. 检查 `before_prompt_build`、`before_tool_call`、`after_tool_call`、`session_end` 四个 Hook 被真实运行时识别；
7. 删除临时状态目录。

成功标志：

```text
[smoke:openclaw] PASS plugin=loaded skill=媒介助手
```

这层主要发现：

- manifest、package entry 或构建产物错误；
- OpenClaw API/Hook 名称不兼容；
- Skill frontmatter 或目录结构错误；
- 源码能编译，但 YP 内置 OpenClaw 无法装载。

### 3.4 合并前：Headless 闭环

```bash
npm run test:headless
git diff --check
git status --short
```

`test:headless` 依次执行 `test:fast` 和 `test:openclaw`。测试结束后检查工作区，正式状态文件不应因测试发生变化：

```bash
git diff -- YPmcn/state/confirmation_guard.json
```

预期无输出。如果出现差异，优先修复测试隔离，不要提交测试生成的会话状态。

## 4. 独立真实 Live E2E

`test:fast` 和 `test:openclaw` 不运行模型，也不访问真实业务链。每个准备交付的业务版本必须由一个未参与实现的 verifier，用一条脱敏真实 Brief 跑完整链路；不能再用“本地脚本模拟了每个 Tool”代替。

历史 Live E2E 状态（2026-07-19，YPmcn `3.1.5`）：`BLOCKED / RANKING DATABASE SCHEMA`。该版本已按 §4.1 从新 tgz 安装到 YP Action `2026.7.1`，Plugin、Skill 和显式配置的 `ypmcn-mcp` 均可用。全新会话 `47d1f3b7-e28d-4a44-b205-7204bdcac4f3` 中，需求 `bb0759a997854e448c68149b16e0aaab`（demand `1784430089674707`，version `1`）真实写入成功，trace `85caa2bd-ba3b-4359-beae-fb33208856a9`。

`search_creators` 真实返回需求人数 1、候选达人 377、供给倍数 377，候选池写入成功但向量召回降级为 `vector_config_missing`；原 Agent Tool trace 为 `41318ff4-48dc-4440-9f3b-9cd04ea954b1`，同一 MCP 的受控核验 trace 为 `6d0bdc7d-7ab6-4471-9696-70a0177f7598`，确认候选涉及 5 家机构。响应仍缺五个供给计划字段。按用户授权，只在临时确认状态补入并标记 `synthetic_test_only`：目标提报 20、预计有效 20、预计缺口 0、推荐 MCN 5、MCN 覆盖 377、人工补充 1、MCN/人工比例 377:1；真实基础量不得改写。OpenClaw 对长结果的截断曾使 Agent 重建出“223 个候选、4 家机构”的错误弹窗，测试者选择停止且未确认。这是旧版本记录；当前供给弹窗作为业务命令执行，不再使用 Hook marker 门禁。

该历史版本的 Hook 随后生成与当前 requirement 和请求指纹绑定的四分区确认弹窗；YP Action 将选择结果返回为“完整问题文本: 确认供给方案”，Hook 已正确识别并在同一轮放行。真实 `rank_mcns` 写入因 pymysql 报告未知列 `mcn_run_id` 返回 `INTERNAL_ERROR`，trace `728bc307-dcd9-46cc-aa4e-4a44e92a22e4`；错误弹窗以“结束 / 重试一次”提供选择。该错误具有确定性，因此选择“结束”，未盲目重试。未生成 MCN 排名，故没有调用字段选择、企微外发、回收、达人排序、CSV 或跨 session 恢复。此处必须报告后端错误并等待 Provider/数据库修复，不能要求用户反复输入“继续”，也不能声称外发完成。

历史 `3.1.6` 只读会话 `55d827cc-9d13-41f8-8c5d-93164e875abd` 复测同类 Brief：`gate=ready`、`missingRequired=[]`、`semanticAmbiguities=[]`、`submissionDeadlineAt=2026-07-20 18:00:00`、`kolOfficialPriceL1=[0,40000]`，没有需求确认弹窗，也没有业务 Tool 调用；旧版 Hook 也拒绝了错误重建的供给确认。当前供给确认不再使用旧 Hook marker；企微外发则使用独立的 AskUserQuestion 弹窗指纹和后续调用参数指纹双重绑定。

当前 checkout 已通过本地全仓库验证、OpenClaw 源码装载和 Provider 只读 `tools/list`；尚未执行新版本的真实企微写入 Live E2E，不得用上述历史会话代替。

### 4.1 宿主切换、版本与安装顺序

Live E2E 必须从本次代码生成的新版本安装包启动，不得直接安装源码目录、复用旧 tgz 或覆盖同版本重打包。如果独立 OpenClaw/Gateway 与 YP Action 同时运行，必须按以下顺序执行：

1. 先关停独立启动的 OpenClaw/Gateway，并确认其进程已退出；这里不是要关闭 YP Action 内置运行时。安装和 E2E 期间不得再并行启动独立 OpenClaw，避免插件缓存、Gateway 和会话状态互相干扰。
2. 将当前版本递增为一个新的 patch 版本，例如 `3.1.5 → 3.1.6`。同步根目录、`YPmcn/` 的 package/lockfile、三份 Plugin manifest 和发布版本测试；版本一致性由 `npm run verify` 确认。
3. 在 Git 根目录执行：

   ```bash
   npm run test:headless
   npm run verify
   npm run pack:yp
   ```

4. 记录 `npm run pack:yp` 最后输出的新 tgz 绝对路径。产物应位于 `packages/releases/`，文件名中的版本必须与本次 patch 版本一致。
5. 保持独立 OpenClaw/Gateway 关闭，在 YP Action 中依次打开“设置 → 插件 → 安装本地路径”，选择上一步刚生成的 tgz，不得选择源码目录或同名旧包。
6. 安装后核对 Plugin ID、新版本号和 Skill。当前 YP Action `2026.7.1` 不会从原生 Plugin 自动导入包内 `.mcp.json`；若“已安装”列表没有 `ypmcn-mcp`，打开“`MCP → 自定义 → + 自定义`”，按包内配置显式创建：

   ```text
   服务名称：ypmcn-mcp
   传输类型：服务器推送事件 (SSE)
   URL：https://mcp.eshypdata.com/sse
   ```

   保存后必须看到 `ypmcn-mcp` 和 `sse · https://mcp.eshypdata.com/sse`；旧的 `YP-server` 或其他历史别名不能作为通过证据。该手工步骤只恢复当前宿主的运行时注册，不代表安装器自动导入已修复。
7. 等 Gateway 因 MCP 配置变更完成重启后新建对话，在新对话中输入本次脱敏真实需求并执行完整 E2E。不得复用安装前的对话上下文。

任一步无法证明时停止后续外发，结果记为 `BLOCKED`。

### 4.2 前置条件

- YP Action 使用按 §4.1 刚安装的待验 tgz，MCP 指向 `https://mcp.eshypdata.com/sse`。
- 开发 MySQL、Qdrant、embedding/rerank 和外部 API 都是测试环境。
- 只选择专用测试机构/企微群，项目名前缀固定为 `E2E-YPmcn-<日期>-<唯一标记>`。
- Brief 必须来自真实需求结构但已脱敏，明确平台、数量、提报截止和单达人预算档位。
- 媒介本人在 UI 中完成供给方案、机构/字段/消息和不可逆外发确认；自动化不得代点。

任一前置条件不满足，结果写 `BLOCKED`，不得切到生产或用 Mock 冒充。

### 4.3 唯一验收链

```text
真实 Brief
→ Agent 三遍解析；缺失/歧义时 AskUserQuestion
→ validate_requirement 成功后同轮立即 search_creators（真实 SQL/Qdrant/rerank provenance）
→ 固定格式展示需求数、实际供给数、供需比与建议拓展数 → Ask 确认
→ rank_mcns（动态 MCN排序数量）→ 取得真实 inquiry_id
→ 若供给确认选择达人拓展，则 manual_source_creators(requirement_id,target_count) 回显同一 inquiry_id
→ 按机构名称展示并由媒介选择
→ select_inquiry_form_fields(url) → 确认字段与企微消息
→ create_with_distributions 本地预检 → 按返回参数弹 AskUserQuestion 多行警告 → “确认发送”后同参数再次调用至测试企微
→ sync_mcn_inquiry_status
→ 用测试表单提交至少一条真实回填
→ sync → ingest_mcn_submissions → sync
→ rank_creators({requirement_id,inquiry_ids?})
→ 目标 Provider：按确认字段导出首批名单 → create_submission_batch
→ 目标 Provider：新建 session，仅用 demand_id + demand_version 调 get_workflow_state 恢复
```

目标 Provider 的完整 Live E2E 必须跑到 CSV 与新 session 恢复；只跑到 `validate_requirement` 或本地 Hook 回放不算该验收。当前 production 的 `create_submission_batch` 和 `get_workflow_state` 入参语义不兼容，安全 E2E 的合格结果是在排序后本地返回 `integration_required`、不发出这两个 Provider 调用；在 Provider 发布目标入参前，不得把 Mock、字段猜测或局部回放冒充完整 Live E2E。

### 4.4 检查点与证据

| 检查点 | 必须看到的真实证据 |
|---|---|
| Brief 解析 | 单达人预算/档位、截止、数量准确；每条原文条件有字段、原文保留或澄清去向 |
| Agent/Hook | 首个业务 Tool 正确；需求确认每题只有一句直接问题，2–4 个选项均有说明且不遗漏独立业务选择；已提交选项在同轮被当作执行命令；本地 JSON 记录实际成功结果后的 phase/next_action |
| Search | MCP 返回真实候选、`retrieval_mode`、provenance 和完整供给事实；界面以固定格式显示供需比与建议拓展数 |
| 外发 | 首次调用返回 `EXTERNAL_SEND_CONFIRMATION_REQUIRED` 且无 Provider 调用；AskUserQuestion 完整显示 MCN、字段和多行消息；只有“确认发送”允许同参数第二次调用，Reject/取消/关闭/超时无 Provider 调用；外部 project/distribution ID、测试企微消息、表单链接与选中机构一致 |
| 回收 | sync 读取真实 distribution；同一 inquiry 被更新而非重复创建 |
| 精排 | 只使用已回收/有效达人拓展候选；硬条件违规为 0；run_id 可查 |
| CSV | `rank_creators` 后立即输出首批名单；表头与实际外发的 `columns[].name` 数量、顺序和文字完全一致，值按对应 `columns[].key` 取得，空值不伪造成 0 |
| 恢复 | 全新 session 恢复同一 phase、ID 和唯一下一步，不依赖旧聊天上下文 |
| 幂等 | 对同一测试写请求做受控重复调用，资源 ID 不变；unknown 不自动重发 |

只记录第一个失败检查点，并保存脱敏 trace/session、Tool 名、Hook 错误码、MCP 错误、相关业务 ID 和期望行为。修复后使用同一 Brief、全新 session 与新的测试项目标记重跑整条链，不能从断点后直接续测。

### 4.5 通过标准

结论只能是 `PASS`、`FAIL` 或 `BLOCKED`。全部检查点通过且 verifier 与实现者不是同一推理链才可写 PASS；任何真实外部环节未运行均不得写“基本通过”。

## 5. 确定性 Hook 与业务链如何验证

当前执行面的确定性门禁位于 `YPmcn/tests/native-hooks.test.mjs`：它验证 Skill/resources/prompts 与非外发 Tool 放行、企微外发 AskUserQuestion 多行正文、弹窗与后续参数指纹绑定、参数变化重新确认、弹窗命令记录、本地 JSON 转换及 shell/curl 绕过阻断。Node Hook 不推进数据库 phase，也不校验 Brief、ID 或普通 Tool 顺序；它只在实际成功结果后更新本地编排投影，因此本地测试不得声称完整业务恢复链已通。

`tests/test_hooks.py` 及 `YPmcn/hooks/*.py` 仅是历史回归工件，已从 `test:fast` 和 `verify` 移除；需要考古旧行为时可显式运行 `npm run test:legacy-hooks`，其旧 phase 不是当前 workflow 证据。

真实业务链只能按 §4 的 Live E2E 验收；本地 Hook 回归不能替代 MCP、开发库、测试企微、回收和跨 session 恢复证据。

## 6. 失败定位顺序

### Fast 构建失败

先单独运行组件测试：

```bash
npm --workspace YPmcn test
npm --workspace vector-mcp test
```

优先处理第一个 TypeScript 或测试错误，不要先打包重试。

### Native Node Hook 失败

单独运行：

```bash
npm --workspace YPmcn test
```

按输出定位：

- AskUserQuestion 被拒绝、取消、关闭或超时：不得执行第二次外发 Tool，也不应有 Provider 写入；
- `YPmcn guard unavailable`：外发 Hook 异常并 fail closed，先修复本地 Hook；
- `INTEGRATION_REQUIRED`：检测到 shell/curl 绕过企微外发 Tool。

### MCP Smoke 失败

```bash
npm run test:mcp
```

- `Non-JSON MCP stdout`：检查 server 是否向 stdout 打印了日志；
- `Timed out waiting for initialize/tools/list`：检查进程启动、异步阻塞和 stdin readline；
- `tools/list returned no tools`：检查 Tool 注册和构建产物；
- server 提前退出：查看 smoke 输出附带的 stderr。

### OpenClaw Headless 失败

```bash
npm run test:openclaw
```

- `plugin ... not loaded`：检查 `openclaw.plugin.json`、`package.json` 和 `dist/index.js`；
- diagnostics 非空：以 OpenClaw 返回的首个诊断为准；
- Skill not eligible：检查 `SKILL.md` frontmatter、目录和运行要求；
- Hook 数量或名称不符：检查 `api.on(...)` 注册名，禁止使用运行时不支持的 Hook；
- CLI 路径不存在：确认 YP Action 已安装，或用 `OPENCLAW_CLI` 指向待测版本。

## 7. 何时运行完整 Verify

日常小改不必反复运行全量验证。以下情况运行一次：

- 合并或发布前；
- 修改 Spec、manifest、发布包内容；
- 修改根安装图、依赖或 workspace；
- 修改安全扫描、provider comparator 或文档同步；
- 相邻测试不足以覆盖风险；
- Headless 出现跨组件失败。

```bash
npm run verify
```

`verify` 包含 Spec、文档同步、安装图、密钥扫描、Plugin/Native Node Hook、provider comparator、Skill 文档、vector MCP 和发布包结构验证。开发 Provider 独立证据由默认只读命令提供；生产路由仅作诊断：

```bash
npm run verify:provider
npm run verify:provider:prod
```

开发机绿色不得描述为生产 provider 已通过。

## 8. YP Action 发布候选冒烟

本节的 UI 冒烟与 §4 Live E2E 使用同一个新版本包。如果独立 OpenClaw/Gateway 与 YP Action 同时运行，先关停独立 OpenClaw/Gateway，再按 §4.1 递增 patch 版本、验证并打包：

```bash
npm run pack:yp
```

使用“YP Action → 设置 → 插件 → 安装本地路径”选择 `packages/releases/` 下刚生成的 tgz。安装成功后按 §4.1 核对或显式创建 MCP，等 Gateway 重启后新建对话再输入需求，不得复用旧对话。YP Action UI 只验证 Headless 无法覆盖的桌面集成面：

1. 安装器能识别正确 Plugin ID 和版本；
2. Plugin、Skill 正常装载，包内只含远程 MCP 配置且不含 Vector MCP；YP Action `2026.7.1` 按 §4.1 显式创建 `ypmcn-mcp`，并把安装器未自动导入记录为宿主限制，不能用安装前已有的全局 entry 冒充自动安装；
3. 新会话能发现“媒介助手”；
4. 输入最小虚构需求后，模型选择正确 MCP Tool；
5. 未确认外发时 Native Hook 能阻断；
6. 正常 Tool 结果能推进会话状态；
7. 重启 Gateway/YP Action 后启用状态和配置仍存在；
8. Gateway 日志没有 unknown hook、plugin load error 或 MCP connection error。

### 8.1 原生多问题弹窗基线

YPmcn `3.1.9` 已在 YP Action 中完成只读 Live 冒烟，会话 `457f15e0-ff39-4818-a46e-e25e676571aa`。验收基线为：同一次 `AskUserQuestion` 渲染 2–3 个独立问题；每题使用简短单句、2–4 个带说明的选项，并保留宿主“其他”输入；四种独立价格口径必须同时显示，不能因 Hook 上限被拒绝后再由模型删减。只读冒烟在检查完页面后关闭弹窗，不提交答案，也不调用业务 Tool。

UI 冒烟失败时，先归类：

- Headless 也失败：回到源码和运行时装载层修复；
- 只有安装后失败：检查 tgz 内容、安装器和配置同步；
- 只有模型选 Tool 错误：检查 Skill 路由、Tool 描述和会话上下文；
- 只有桌面交互失败：检查 YP Action callback/UI 层；
- 只有生产调用失败：运行只读 provider 检查，禁止用 Mock 结果替代。

## 9. 安全与清洁要求

- Headless 测试不得复用或修改 `~/Library/Application Support/YP Action` 下的正式状态。
- `verify:provider` 与 `verify:provider:prod` 只发送 `initialize`、`notifications/initialized`、`tools/list`；自动测试不得调用生产业务 Tool。
- CI/Headless 自动测试不得调用外发、写数据库或不可逆 Tool；§4 Live E2E 只在专用测试环境由媒介显式确认后执行。
- 测试状态必须写入临时目录，结束后自动删除。
- 不把日志、客户 Brief、完整 payload、凭据或未脱敏响应写入仓库。
- 不直接编辑 `dist/`、`packages/.staging/` 或 tgz。
- 失败测试必须报告 FAIL；未运行必须报告 NOT RUN。

## 10. 最短开发循环

常规修改：

```bash
npm run test:fast
git diff --check
```

Plugin、Skill 或 OpenClaw 修改：

```bash
npm run test:headless
git diff --check
```

发布候选：

先按 §4.1 关停并行的独立 OpenClaw/Gateway，并递增为未使用过的 patch 版本，再执行：

```bash
npm run test:headless
npm run verify
npm run pack:yp
```

最后只在 YP Action 中执行一次发布冒烟，不再把桌面 UI 当作首轮调试器。
