# 给执行 Agent 的工作说明

这份文档解释在本仓库里怎样安全地完成一个任务。它不是业务规则本身；业务规则以根目录 [`spec/`](../spec/README.md) 为准。开始前先读[文档导航](README.md)，了解“批准契约、本地代码、远端 Provider”是三种不同证据。

## 1. 先判断自己在改什么

| 任务例子 | 是否需要先改 Spec | 通常先读什么 |
| --- | --- | --- |
| 把文档中旧的 `target_count` 改为当前 `size` | 否 | 对应工具卡、运行时审计、链接检查 |
| 增加 Tool 参数、改错误码或返回字段 | 是 | Change Proposal、`spec/mcp.json`、Provider 契约 |
| 改变工作流能否跳步、恢复依据或 Hook 阻断条件 | 是 | `spec/workflow.json`、`spec/hooks.json`、相邻测试 |
| 修正本地实现与已批准 Spec 不一致 | 通常否，但必须确认 | Spec、现有测试、最小复现 |
| 改排序公式 | 不可自行决定 | `spec/algorithms.json`；当前为 `external-unverified`，先阻断并请求批准 |

不要因为现有代码、聊天记录或旧文档“看起来可以”就发明规则。正式优先级是：

```text
安全与数据完整性
> 已批准 Spec
> 已批准 Change Proposal
> 可重复的测试 / 审计证据
> 当前实现与历史文档
> Agent 推断
```

## 2. 当前运行边界（必须知道）

- 插件有 4 个 Node Hook：`before_prompt_build`、`before_tool_call`、`after_tool_call`、`session_end`。它们做确定性提示、守卫、记录和清理，不拥有业务数据库事实。
- 本地 workflow JSON 是**编排投影**：它可以告诉下一步建议做什么，不能证明 Provider 写入、机构回填或企微外发成功。
- Host 侧工具名是 `mcp__ypmcn__<bare-tool>`；远端 `tools/list` 使用 bare tool name。不要把二者误当作两个接口。
- `create_with_distributions` 当前每次都先经过本地 `AskUserQuestion` 确认。最新未过期的确认回执可跨 turn，只放行下一次调用一次；当前消费回执时不再次比较调用参数，所以绝不能把它当作长期或通用授权。它更不是发送成功证据：只有实际响应中的逐机构发送明细可以证明外发成功；`sync_mcn_inquiry_status` 不能倒推出发送成功。
- 远端 Provider 当前与批准契约存在已知硬差异。任何涉及 `create_submission_batch` 或 `get_workflow_state` 的任务，先读[运行时审计](MCP_TOOL_RUNTIME_AUDIT_2026-07-23.md)，不要为了“跑通”转换或猜测字段。

## 3. 动手前的最小任务单

每个实现任务先把下面六件事写清楚；信息缺失且会改变范围时，先停下来补齐，而不是靠猜测扩张任务。

```yaml
goal: "要解决的可观察问题"
allowed_paths:
  - "可修改的精确路径或目录"
do_not_touch:
  - "明确不属于本任务的路径"
acceptance:
  - "完成后能观察到什么"
verification:
  - "要运行的最小命令"
rollback_or_risk: "失败时如何撤回，或为什么不能撤回"
```

涉及 Tool、字段、外部写入、权限、Hook、流程阶段、错误码、数据库不变量或算法时，再补上关联 Change Proposal、Impact Analysis 与批准的 Spec 版本。

## 4. 推荐执行顺序

1. 阅读任务单、相关 Change Proposal、Spec、实现和测试；先确认当前工作树是否已有别人的改动。
2. 用只读检查定位真实问题，例如 `rg`、已有测试、Provider 只读契约检查。不要先改一大片再找原因。
3. 只修改验收所需的文件。清理本次产生的死代码、临时文件和调试痕迹，但不顺手重构无关模块。
4. Bug 修复先建立稳定复现，再做最小修复；契约变化覆盖正常、缺失、非法、外部副作用、错误映射和未知结果对账。
5. 运行任务单中的最小直接验证。失败就修复后重跑；没有运行的项目要如实写 `NOT RUN` 和原因。
6. 最后检查 diff、交叉链接和文件边界，说明剩余外部风险。

示例：修复“手工拓展参数名过时”的文档任务时，先从工具参考和运行时审计确认实际为 `size: "10"`，再修改说明和示例，运行 `npm run verify:docs` 与链接检查；不应同时改 MCP Schema 或强行调用远端写工具。

## 5. 并行与文件边界

并行只有在任务之间没有输入依赖、不会改同一文件、各自有独立验证时才有意义。优先使用独立 worktree；若执行环境明确要求共享工作树，必须先分配不重叠的文件，并在写入前检查工作树变化，绝不覆盖别人的未提交内容。

下列依赖通常仍应串行：

```text
Change Proposal → Spec → 数据库 / MCP → Hook / Skill → 集成测试 → 打包
```

## 6. 验证门禁怎么读

| 命令 | 回答什么 | 不能证明什么 |
| --- | --- | --- |
| `npm run test:fast` | 插件与本地 Hook 的快速回归 | 远端 Provider 可用或真实外发成功 |
| `npm run test:openclaw` | 源码插件是否能被内置宿主加载 | 生产桌面交互与远端业务数据 |
| `npm run verify` | 离线契约、测试、打包等仓库门禁 | 线上环境与外部写入成功 |
| `npm run verify:docs` | 自动生成事实区块是否跟 Spec 一致 | 叙事文档是否过时 |
| `npm run verify:provider:prod` | 当前远端输入契约是否兼容 | 写路径成功、稳定 outputSchema 或 E2E 成功 |

当前 Provider 检查失败会暴露已知差异，不应为了让命令变绿而放宽本地契约。外部服务验证只读优先；需要真实写入、外发或读取真实业务数据时，必须有明确授权、隔离测试对象和清理方案。

## 7. 什么时候必须阻断

输出 `BLOCKED` 或请求方向，而不是继续猜测，至少包括：

- Spec 缺失、互相冲突，或要改的算法仍是 `external-unverified`；
- 任务需要生产凭据、真实客户/达人数据、不可逆迁移或外发，但没有明确授权；
- Provider schema 与批准契约不兼容，且没有安全映射；
- 要修改的文件正被其他任务占用，或工作树改动归属不清；
- 无法提供合理的验收条件、验证命令或回滚说明。

“测试还没跑完”“问题比较复杂”不等于阻断；此时应继续在已授权范围内排查并如实报告。

## 8. 完成报告格式

不要伪造成功、测试或文件列表。完成时使用以下四个字段，列出实际内容：

```markdown
- changed_files: 实际变更的仓库相对路径列表
- fixed_issues: 已修复的问题列表
- known_risks: 仍需外部验证、未运行检查或无法消除的风险
- task_report: 做了什么、运行了什么、结果如何
```

若任务涉及外部 Provider，报告必须把“本地通过”“远端兼容”“真实业务成功”分开写；三者不能互相替代。
