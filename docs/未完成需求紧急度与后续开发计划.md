# 未完成需求紧急度与后续开发计划

日期：2026-07-16

依据：

- [完整业务需求与测试 Checklist](完整业务需求与测试Checklist.md)
- [业务需求实际完成情况核验](业务需求实际完成情况.md)
- `spec/manifest.json` 指向的正式契约

## 1. 总体判断

当前不能直接进入“补页面、接企微、做体验优化”阶段。最紧急的问题是：**正式业务规则仍有缺口、统一验证失败、生产数据库与 Provider 没有权威实现证据、排序算法未批准。** 如果这些基础问题不先解决，后续集成会反复修改数据模型、接口和状态机。

建议按四级处理：

| 紧急度 | 定义 | 处理原则 |
| --- | --- | --- |
| **S0｜立即阻断** | 不解决就无法安全继续开发、集成或判断结果正确性 | 立即处理；完成前不接生产流量，不扩展下游功能 |
| **S1｜首版上线阻塞** | 构成 P0 业务闭环或数据安全边界 | S0 后按依赖顺序实现；全部通过才允许首版验收 |
| **S2｜首版质量与运营必需** | 不一定阻断最小链路，但会造成不可运营、不可复核或高人工成本 | 首版发布前完成，或由业务负责人书面接受延期 |
| **S3｜待决策/规模化** | 依赖业务口径、外部授权或容量目标 | 先确认决策，再立独立 Change；禁止凭经验写死 |

这里的“完成”必须同时满足：正式 Spec 已确定、仓库实现已测试、生产数据库/Provider 已有只读或集成证据、对应业务 TC 已按真实样本执行。仅有 Schema、Skill 文档或 Hook 本地测试不算完成。

## 2. S0｜立即阻断

### S0-1 恢复可信开发基线

关联：仓库统一验证；间接影响全部 BR。

当前问题：

- `npm run verify` 在 root workspace installation 阶段失败。
- `YPmcn/package-lock.json` 记录版本 `3.0.2`，当前包版本为 `3.0.3`。
- 工作区已有大量未提交修改，必须避免把业务实现建立在未知基线上。

开发任务：

1. 确认 `3.0.3` 是否为目标发布版本。
2. 使用根 workspace 的 npm 流程更新锁文件，禁止手工改锁文件版本字段。
3. 运行一次完整 `npm run verify`，记录真实通过项和失败项。
4. 清点当前删除/新增的 Hook、reference-mcp 和文档变化，确认它们属于同一版本边界。
5. 统一 README、插件 manifest、package 与 lockfile 的版本及运行时描述。

验收：

- `npm ci` 可从根目录复现安装。
- `npm run verify` 返回 0。
- `git diff --check` 通过。
- 发布包不包含 mock、测试、凭据、绝对路径或过期 Spec。

### S0-2 补齐三项核心业务契约

关联：BR-011、BR-014、BR-017；TC-003、005、012、013。

当前问题：

- 未正式定义并实现“账号类型与内容标签 OR”。
- 未建立 MCN、野生达人、去重总量三项资源统计。
- 未建立客户目标、机构提报、内部候选池三层数量模型。

开发任务：

1. 在 requirement constraint grammar 中明确账号类型/内容标签的 OR 表达和缺失值语义。
2. 为资源来源定义稳定枚举，例如机构、野生、重复合并后的逻辑身份；不得仅靠展示层计算。
3. 为三层数量定义独立字段、单位、约束、倍率来源、版本与审计行为。
4. 规定跨机构、跨来源同一达人如何基于 `(platform, kw_uid)` 去重。
5. 明确倍率变化只生成新计算版本，不改写客户最终目标。
6. 先修改正式 Spec 和 Change Proposal，再修改数据库、Provider、Skill 和测试。

验收：

- TC-003、005、012、013 有自动化回归。
- OR 三种输入组合均通过：仅类型、仅标签、两者同时提供。
- 未知字段不会被当成“不符合”，也不会自动当成命中。
- 三层数量和三类资源统计可同时查询、导出并审计，字段不复用。

### S0-3 确立服务端权威状态与数据库不变量

关联：BR-001～005、007、019、023～025、033、035、038、040。

当前问题：

- 当前 Hook 只维护可丢失的本地会话投影，不能授予生产动作。
- `spec/database.json` 的实体、唯一约束、快照、审计和幂等仍是 `external-unverified`。
- 发送、恢复、迟到数据和 offer promotion 缺少生产并发证明。

开发任务：

1. 实现 migration：主键、唯一键、scope、外键/逻辑 join、状态版本和不可变审计表。
2. 服务端持久化 `state_version`、`allowed_actions` 和稳定 correlation key。
3. 实现 snapshot append-only、late data 不改冻结结果、offer promotion 追加版本。
4. 对发送和 recovery 使用事务、CAS 或行锁，保证单一 owner 和幂等。
5. Hook 保留 deny-only 职责；任何本地状态都不得越权授权服务端动作。
6. 为 14 项数据库/后端不变量逐项补集成或并发测试。

验收：

- 重复提交、并发恢复、未知发送结果、迟到数据和重复 offer promotion 均 fail closed 或返回同一幂等结果。
- 所有冻结结果可通过 snapshot/hash 复现。
- 审计纠正只追加新事件，不更新或删除历史记录。
- TC-016、018、020 的服务端路径通过。

### S0-4 批准排序与筛选算法边界

关联：BR-015、016、020、021、025、026；TC-002、004、006、007、011、014。

当前问题：

- `spec/algorithms.json` 仍为 `external-unverified`。
- 没有批准的排序因素、权重、缺失值处理、返点排序开关和解释格式。
- 向量组件的局部逻辑不能自动升级为正式业务算法。

开发任务：

1. 冻结硬条件字段、操作符、边界包含关系和 missing policy。
2. 明确返点先硬筛、后作为可配置排序因素的两段语义。
3. 定义排序因素、权重版本、开关、归一化、平分规则和解释字段。
4. 明确平台真实值与其他来源冲突时的展示值、计算值和复核状态。
5. 建立算法 golden cases，覆盖零值、缺失、异常、过期、冲突和同分。
6. 算法批准后再实现 Provider；禁止从现有向量代码反推业务权重。

验收：

- 同 requirement snapshot、数据 snapshot 和 algorithm version 得到可复现结果。
- 修改排序权重不改变硬筛集合。
- 缺失不当零、零值不当缺失、异常数据不静默进入高排名。
- 每个结果能解释硬筛结论、排序因素、来源和数据版本。

## 3. S1｜首版上线阻塞

### S1-1 Brief 解析、确认与版本闭环

关联：BR-002、006～010；TC-001、010、011、018。

依赖：S0-2、S0-3。

开发任务：

1. 实现自然语言、原始 Brief、语音转写文本的统一输入入口。
2. 保存原文、规范化值、不确定项、解析版本和解析器版本。
3. 将字段分为必填、建议、软性和待确认，并允许按项目配置。
4. 实现媒介编辑、确认、撤回确认和新版本生成。
5. 硬条件或方向变化新建需求版本/批次；软性补充按批准规则处理。
6. 下游只能读取最新已确认 snapshot，不读取草稿或本地会话推断。

验收：

- 模糊单值、左右表达、缺失字段和冲突 alias 均进入确认或明确失败。
- 未提及字段保持空/未知，不产生模型臆填。
- 修改前后值、操作者、时间和原因完整可查。
- TC-001、010、011、018 通过。

### S1-2 硬筛、供给判断与不足阻断

关联：BR-011、013～019；TC-002～005、012、013。

依赖：S0-2、S0-4、S1-1。

开发任务：

1. 从已确认 requirement snapshot 生成硬筛执行计划。
2. MySQL 作为权威数据源，向量只参与软召回/特征，不覆盖硬条件。
3. 输出每个候选的命中、未命中、缺失和无法判定项。
4. 计算机构、野生、去重总量及三层目标数量。
5. 先输出硬筛供给，再判断充足/不足；不足时持久化缺口并停止排序扩散。
6. 复跑绑定 requirement、source 和 algorithm 三类版本。

验收：

- 硬筛未通过的达人不会进入排序池。
- 同一输入与三个版本相同可复现。
- 供给不足不自动降低门槛、不自动补野生、不修改客户目标。
- TC-002～005、012、013 通过。

### S1-3 MCN 优先与野生补充闭环

关联：BR-018、028～030；TC-008、013、015。

依赖：S1-2、S0-3。

开发任务：

1. 资源不足时先创建机构提报批次，保存目标、截止时间、供应商和状态。
2. 分别统计已邀请、已响应、已提报、有效、选中、拒绝和过期。
3. 提报不足后生成缺口，不自动启动野生补充。
4. 取得当前服务端状态绑定的人工确认后，才允许 `manual_source_creators`。
5. 将机构和野生来源统一晋升为版本化 supply offer，同时保留原始来源。
6. 供应商删除/补充/确认必须追加记录并可被内部复核。

验收：

- 未完成机构阶段或缺少确认时，野生补充被服务端拒绝。
- 同一达人跨机构/来源只计入一次去重总量，但所有来源均可查。
- 重复摄取不生成重复逻辑提报。
- TC-008、013、015 通过。

### S1-4 推荐版本与人工调整

关联：BR-020～023；TC-006、019。

依赖：S0-4、S1-2、S1-3。

开发任务：

1. 自动推荐持久化为不可变 run，绑定全部输入版本。
2. 实现保留、删除、补充、调序、改数量及原因。
3. 每次人工调整生成新版本，不覆盖自动结果。
4. 支持撤销到上一版本和恢复自动版本。
5. 只有已确认推荐版本可进入发送或客户提报。
6. `audit_manual_adjustment` 必须校验 operator、run、调整对象和状态版本。

验收：

- 自动结果与所有人工版本都可回看和比较。
- 未确认版本无法被下游读取。
- 调整记录含前后值、操作者、时间、原因和版本。
- TC-006、019 通过。

### S1-5 企微生成、预览、确认、发送与回执

关联：BR-032～035；TC-009、015、020。

依赖：S1-1、S1-3、S0-3。

开发任务：

1. 明确首版企微账号、授权方式、群类型和模板字段。
2. 从已确认 Brief snapshot 生成消息和表单字段；缺失项留空或提示。
3. 预览内容持久化为 selection result，编辑后生成新版本。
4. 服务端校验 operator 权限、三项确认、字段 snapshot 和当前状态。
5. 建立 send operation、idempotency key、provider correlation key、回执和失败原因。
6. 未知写结果只允许查询/对账，不允许直接重发。
7. Tool 表面继续禁止自动拉群、加人、踢人。

验收：

- 严格执行生成 → 预览 → 人工确认 → 发送。
- 未确认、字段漂移、状态漂移、权限不足时均被服务端拒绝。
- 成功、失败、未知三种结果可查且不会重复发送。
- TC-009、015、020 通过。

### S1-6 RBAC、审计与复核队列

## 4. S2｜首版质量与运营必需

## 6. 推荐实施波次

| 波次 | 工作内容 | 退出条件 |
| --- | --- | --- |
| Wave 0 | S0-1 开发基线；S0-2 核心业务契约 | 统一验证通过；OR、资源分层、三层数量 Spec 冻结 |
| Wave 1 | S0-3 数据库/状态权威；S0-4 算法批准 | migration、并发测试和 algorithm golden cases 通过 |
| Wave 2 | S1-1 Brief；S1-2 硬筛与供给 | TC-001～005、010～013 通过 |
| Wave 3 | S1-3 MCN/野生；S1-4 推荐调整 | TC-006、008、013、019 通过 |
| Wave 4 | S1-5 企微发送；S1-6 RBAC/复核 | TC-009、015、017、018、020 通过 |
| Wave 5 | S2 标签质量、运营、导出和失败接管 | TC-007、014、016、021 通过 |
| Wave 6 | S3 采集、部署、性能、备份 | 已确认非功能目标及压测/恢复演练通过 |

不得并行的关键依赖：

- 三层数量模型未冻结前，不做供给缺口页面或报表。
- 服务端状态权威未完成前，不接真实发送和自动恢复。
- 排序算法未批准前，不开发业务权重配置 UI。
- RBAC 未完成前，不开放原始 Brief、达人详情和导出给多角色。

可以并行的工作仅限 write set 不重叠且契约已冻结，例如：数据库 migration 与算法 golden cases；前端原型与后端实现也只能在字段、状态和错误码冻结后并行。

## 7. 每个开发任务的交付要求

每个任务开始前固定以下边界：

```yaml
goal: "单一可验证结果"
allowed_paths: ["明确允许修改的路径"]
forbidden_paths: ["其余路径"]
acceptance: ["对应 BR/TC 的可观察结果"]
verification: ["相关测试", "git diff --check", "修改范围检查"]
```

每个任务交付必须包含：

1. 对应 BR、TC 和 Change Proposal。
2. Spec 变化或“无需变化”的明确结论。
3. 最小实现 diff。
4. 自动化测试结果；未运行必须标记 `NOT RUN`。
5. 数据库/Provider 外部证据边界。
6. 已知风险和回滚方式。

涉及数据库迁移、生产 Provider、企微真实发送、权限或不可逆副作用的任务按 Critical 处理：隔离 worktree、一个写者、独立只读验证、人工批准外部副作用。

## 8. 首版最终完成标准

只有同时满足以下条件，才能把《完整业务需求与测试 Checklist》的首版描述为完成：

- 所有 P0 BR 在生产等价环境可观察，P1 延期项有书面批准。
- OR 标签、两阶段筛选、三层数量、三类资源、来源冲突和人工门禁全部有真实数据证据。
- 生产数据库不变量、Provider 契约、算法版本和服务端状态权威均通过独立验证。
