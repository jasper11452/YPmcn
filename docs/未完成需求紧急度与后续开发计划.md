# 未完成事项与后续开发计划

> 更新：2026-07-23 · 插件包版本：3.4.25
>
> 当前基线：`mvp-v2`。具体包版本以根与 `YPmcn` 的 package manifest 为准。
>
> 总原则：先让“事实能对上”，再扩展功能；当前生产状态为 **NO-GO**。

这不是历史缺陷清单，也不是承诺日期表。它把现在已确认的阻塞、需要外部协作的事项和可验证的完成标准放在同一条依赖链上。详细当前状态见 [集成与上线状态](integration-readiness.md)。

## 排序方式

| 优先级 | 含义 | 处理要求 |
| --- | --- | --- |
| P0｜阻断 | 不解决就不能安全完成主链路或判断写入是否正确。 | 不接生产流量；先修契约和服务端事实。 |
| P1｜首版必需 | 不一定阻止单步演示，但缺失会让业务不可恢复、不可审计或不可运营。 | 在首版验收前完成。 |
| P2｜需业务决策 | 需要产品口径、权限或容量目标，仓库不能凭空决定。 | 先记录决策，再改契约和实现。 |

每项“完成”都至少需要：批准的 Spec、实现和自动化测试、实际部署环境的证据、以及与该项匹配的端到端样本。只有文档、Schema 或本地 Hook 测试，不算完成。

## P0：先消除契约与恢复阻断

### P0-1 对齐远端 MCP 的两个硬差异

**现状（已验证）**：2026-07-23 的只读检查发现 15 个工具可见，但 `create_submission_batch` 和 `get_workflow_state` 的远端输入与本地批准契约不兼容，`npm run verify:provider:prod` 因此失败。

| 工具 | 当前远端输入 | 目标输入 | 需要 Provider 做什么 |
| --- | --- | --- | --- |
| `create_submission_batch` | `requirement_id`、整数 `size`、`submission_batche_page`、`columns` | `requirement_id`、字符串 `size`、`number` | 发布批准的输入 schema，或经过 Change Proposal 共同更新契约；不能让插件猜字段映射。 |
| `get_workflow_state` | `requirement_id` | `trace_id`，或 `demand_id + demand_version` | 支持批准的恢复身份，或先走正式变更流程；不能由客户端推导 requirement ID。 |

**验收标准**：

1. Provider 在目标环境发布新 schema。
2. `npm run verify:provider:prod` 返回 0，且保存当日 schema 快照。
3. 对每个工具用隔离数据取得一次成功和一次明确失败样本，确认失败不会产生写入。
4. 删除临时兼容逻辑，而不是长期保留两套猜测字段。

**例子**：若远端坚持使用 `submission_batche_page`，这不是“客户端补一个 1”即可解决的 UI 差异；它改变了导出的批次语义，必须明确批次从哪里来、如何幂等、怎样审计。

### P0-2 建立服务端写入幂等与未知结果对账

**现状（已验证于 Spec）**：`spec/database.json` 仍标记持久化写入幂等、权威询价同步和选中 MCN 覆盖快照为 `not-implemented` 或远端未验证。外部创建响应丢失时，当前没有已批准的安全恢复查询。

**工作项**：

1. 给每个业务写入定义稳定的幂等键、请求摘要和资源引用。
2. 让服务端能按该键返回“未执行 / 已成功 / 已失败 / 结果未知”，而不是仅返回泛化成功文本。
3. 对外部创建保存可对账的项目/分发身份；事务失败时可确认回滚。
4. 把未知结果固定为停止和对账，禁止客户端自动重发。

**验收标准**：相同请求重复执行不会生成重复业务记录；相同键但不同内容明确报冲突；模拟超时后可通过只读查询得到唯一结果；并发请求不会出现两个有效发送。

### P0-3 让 MCN 选择、发送、同步和回收共享同一事实链

**现状（已验证于流程契约和本地实现）**：本地 Hook 可以记录编排，也已经为每一次 `create_with_distributions` 提供一次性“企微外发确认”：先阻断、经明确确认后仅放行下一次调用一次；取消、拒绝或超时不会发出调用。确认状态 `popup_required`、`approved`、`in_flight`、`consumed` 保留在本地投影中，宿主没有会话上下文时使用插件全局回执兜底。它仍不拥有 Provider 事实，且当前不会因后续参数变化重新核对回执。`rank_mcns` 的机构选择、去重覆盖和 `inquiry_id` 尚没有远端写入验证；`sync_mcn_inquiry_status` 也不能反推企微是否已发送。

**工作项**：

1. `rank_mcns` 在同一事务中落下选中供应商、`(platform, kwUid)` 去重覆盖、风险/倍率和真实 `inquiry_id`。
2. `create_with_distributions` 返回逐供应商、可关联的明确状态；只有 `sent` 才是发送证据。
3. `sync_mcn_inquiry_status` 只同步和维护询价事实，不创建或覆盖发送成功结论。
4. 让回收、排序和新会话恢复都读取上述权威记录。

**验收标准**：用一个隔离需求选择两家供应商；供应商 A 成功、B 失败时，系统只把 A 当作已发送，后续同步也不会把 B 误标为已发送。

## P1：补齐可验收的业务闭环

### P1-1 数据库部署与约束复验

`development-observed` 表示“曾在开发环境观察过”，不是“已经部署”。需要在目标环境重新核对：达人身份 `(platform, kwUid)`、需求与供应商关联、唯一索引、快照与审计记录、事务边界和迁移版本。

**验收标准**：保存当日 schema/索引证据；重复、跨需求和跨供应商的非法关联全部失败；并发与回滚测试可复现。

### P1-2 明确并验证排序算法

`spec/algorithms.json` 当前是 `external-unverified`。先由业务决定硬筛字段、范围边界、缺失值语义、返点/价格的作用、排序权重、同分规则和结果解释，再实现并冻结算法版本。

**验收标准**：同一需求快照、数据快照和算法版本得到同一排序；改权重不会改变硬筛集合；缺失值不被静默当作零；每条结果可解释来源和版本。

### P1-3 形成可控的真实链路验收

建立隔离测试租户、专用需求/供应商/达人、无真实影响的外发通道和清理方案。一次验收至少覆盖：

```text
需求校验 → 搜索或人工拓展 → MCN 选择 → 字段选择 → 企微外发确认 → 一次发送 → 同步/回收 → 排序 → 导出 → 新会话恢复
```

人工拓展路径的例子：先 `validate_requirement`，再用刚返回的 `data.id` 调 `manual_source_creators`；如果该需求已经启动搜索，则先完成 MCN 分支，不能跳步。外发验收还应覆盖：确认回执跨回合仅能消费一次、拒绝/取消/超时不调用 Provider，以及即使同步成功也不把没有 `sent` 明细的供应商写成已发送。每一步保存请求、实际响应、数据版本、操作者、时间和清理结果。

### P1-4 发布与运行安全

完成凭据轮换、发布包密钥扫描、宿主插件来源/权限配置和灰度观测。重点指标包括重复写入、未知发送、状态冲突、迟到数据和恢复失败。

## P2：需要先做业务决定的事项

以下项目目前不应由实现人员自行猜测：

| 决策 | 为什么需要先决定 | 产物 |
| --- | --- | --- |
| 账号类型与内容标签如何组合 | OR/AND 和缺失值会直接改变搜索集合。 | 已批准的约束语法、样例和回归用例。 |
| 客户目标、MCN 提报、内部候选池如何计数 | 三层数量不能复用一个字段。 | 单位、来源、去重规则、倍率和审计口径。 |
| 排序权重与解释格式 | 影响推荐公平性和可复现性。 | 算法版本、golden cases、变更审批。 |
| 企微模板、确认刷新、失败重试和权限 | 直接影响外部发送和合规；当前本地回执不会因后续参数变化自动重新核对。 | 模板、RBAC、变更后是否必须重新确认的规则、重试/对账规则、审计责任人。 |
| 性能、容量和保留期 | 没有目标就无法判断“够不够用”。 | SLO、容量测试和备份/保留方案。 |

## 推荐执行顺序

```text
P0-1 远端 schema 对齐
  └─ P0-2 写入幂等与对账
      └─ P0-3 MCN/发送/同步事实链
          └─ P1-1 数据库部署复验 + P1-2 算法批准
              └─ P1-3 隔离 E2E
                  └─ P1-4 发布与灰度
```

这条顺序不是形式要求：如果先做导出界面或发送体验，随后 schema、幂等或状态事实改变，前面的集成很可能需要重做。

## 每次变更后的最小复查

| 变更类型 | 最小检查 |
| --- | --- |
| 文档或 Spec | `npm run verify:docs` 与 `npm run verify:spec` |
| 本地插件 | `npm run test:fast` |
| Provider 契约 | `npm run verify:provider:prod`，保存当次结果 |
| 有写入影响的 Provider 改动 | 隔离环境 E2E、幂等/并发/回滚证据，以及人工复核 |

历史版本、历史 trace 和旧环境报错只用于设计复测，不会自动变成本计划的当前事实；需要重新执行后才可纳入验收。
