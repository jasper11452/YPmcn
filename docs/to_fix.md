### 两个必须坚持的安全规则

1. **不猜字段。** `create_submission_batch` 的 `submission_batche_page` 与 `columns` 不可以为了“先跑通”而临时拼出来。
2. **不盲重试。** 外部写入的响应未知时，先通过服务端权威记录对账；无法对账就停住，不再次发送。

## 当前实现边界

### 本地 Hook 能做什么

当前 Node 插件有四个 Hook，并会做需求主键形状检查、人工拓展的一次性绑定、企微外发的一次性人工确认、实际结果记录和 shell/curl 绕过拦截。它可以减少调用错误，但不能替 Provider 确认数据库写入、发送成功、服务端权限或幂等。

### 对外发送如何判定

`create_with_distributions` 不会被插件直接放行。每次尝试（包括单个供应商的兜底尝试）都会先被本地 Hook 阻断，要求精确的“企微外发确认” `AskUserQuestion`；只有明确“确认发送”才创建本地放行回执。取消、拒绝、关闭、超时或弹窗错误都不会继续调用 Provider。

最新未过期回执可跨 `AskUserQuestion` 的对话回合存在，并且只放行下一次调用一次。确认过程的 `popup_required`、`approved`、`in_flight`、`consumed` 状态写入本地工作流投影；宿主没有会话上下文时，插件使用无会话全局回执兜底。当前实现不因下一次调用的参数变化重新核对回执，因此参数改动本身不保证会重新弹出确认。

本地确认只证明“允许插件尝试调用”，不是送达凭据。只有该工具的实际成功响应中出现可关联供应商的明确 `sent` 状态，才能写“已发送”；`sync_mcn_inquiry_status` 不能把同步事实变成发送成功。

例如：请求给供应商 A、B 发送，响应只明确 A 为 `sent`，则后续同步只能把 A 视为发送成功；不能因为同步成功或总响应成功，就把 B 也写成已发送。

### 人工拓展达人如何避免串单

每次调用 `manual_source_creators` 前，都要重新 `validate_requirement`，并只使用刚返回的 32 位 `data.id` 调用一次：

```text
validate_requirement → data.id → manual_source_creators({ requirement_id: data.id, size: "20" })
```

数字 `demand_id`、旧会话 ID 和猜测出来的 ID 都不能使用。若先前已启动搜索，需先完成 MCN 分支、字段选择、发送和同步。

## 需要复现后才能重新立项的历史线索

旧版文档记录过向量配置缺失、字段选择器地址不可达、OpenClaw Gateway/能力差异、SQL 列漂移、特定搜索返回 0 等现象。这些记录对排查有价值，但当前没有逐项重测证据，不能写成“仍存在”或“已经修好”。

若要恢复其中某项，复测记录至少应包括：

- 当前插件和 Provider 版本、endpoint、时间；
- 无敏感数据的最小请求与完整脱敏响应；
- 是否产生写入；
- 与 `spec/mcp.json`、`spec/workflow.json` 的对应关系；
- 能重复执行的检查命令和预期结果。

## 推荐处理顺序

1. 先修 B-01、B-02，恢复批准的输入和安全恢复能力。
2. 再修 B-04、B-05，把写入、发送和回收变成可对账的服务端事实。
3. 随后完成 B-03、B-06，稳定输出、数据库约束和算法版本。
4. 最后用隔离环境跑完整链路，并按当天证据更新上线状态。
