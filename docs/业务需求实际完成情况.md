# 业务需求完成度：当前盘点

> 更新：2026-07-23 · 插件包版本：3.4.25
>
> 适用范围：本仓库、当前远端 MCP 广告契约，以及已有的开发环境观察。
> 结论：**有可运行的本地编排和已批准的目标契约，但没有完整生产业务闭环的验收证据。**

此前版本引用的“完整业务需求与测试 Checklist”不在当前仓库中，不能再把其中的编号、通过率或历史版本号当作当前结论。本页改以仓库现存 Spec、实现和 2026-07-23 的远端只读审计为准。

## 怎么读“完成”

为了避免“写了文档”被理解为“已经交付”，本页只使用三种状态：

| 状态 | 含义 | 例子 |
| --- | --- | --- |
| 已有本地能力 | 仓库内有实际实现和可运行的本地验证。 | Node 插件注册了四个 Hook，并记录本地编排状态。 |
| 已有目标契约 | Spec 说清了目标行为，但 Provider、数据库或完整链路尚未验证。 | 需求、搜索、MCN、发送、回收和导出的工作流。 |
| 尚未验收 | 关键服务端事实、成功样本或真实端到端证据缺失。 | 写入幂等、恢复、算法结果、企微发送与新会话恢复。 |

“已有本地能力”不等于“可上生产”。总体生产状态仍以 `spec/requirements.json` 的 `productionReadiness: NO-GO` 为准。

## 当前插件实际负责什么

当前执行面是 Node 插件，不是旧的 Python Hook。它注册四个 Hook：

- `before_prompt_build`：补充本轮需求和本地编排上下文。
- `before_tool_call`：做少量安全守卫，例如需求主键形状、一次性人工拓展绑定、企微外发的一次性人工确认，以及阻止用 shell/curl 绕过声明的 Provider 写工具。
- `after_tool_call`：只根据实际工具结果更新本地编排记录。
- `session_end`：清理过期的本地记录。

它不负责证明远端成功。外发确认只让**本地插件**放行一次调用，不应替代数据库事务、Provider 参数校验、服务端权限或幂等。

### 例子：企微外发的一次性确认

当流程准备调用 `create_with_distributions` 时，Hook 会先阻断工具调用并要求“企微外发确认”。用户明确选择“确认发送”后，最新未过期的本地回执会跨 `AskUserQuestion` 的对话回合保留，并且只放行下一次调用一次；本地投影依次记录 `popup_required`、`approved`、`in_flight`、`consumed`。

取消、拒绝、关闭或超时不会发送。宿主没有会话上下文时，插件使用无会话全局回执来完成同样的一次性门禁，而不会套用其他会话的确认。当前实现也不会因为下一次调用改了参数而重新核对回执，所以不要把“参数改了”误解成“系统一定会再弹一次确认”。

无论本地回执状态如何，只有 Provider 返回按供应商关联的明确 `sent` 结果，才可写“企微已发送”。后续 `sync_mcn_inquiry_status` 成功，只说明同步完成，不能补成发送成功。

### 例子：人工拓展达人

正确的最小步骤是：

1. 用当前客户 Brief 调用 `validate_requirement`。
2. 从真实成功响应里取 `data.id`，例如 `a1b2...` 这样的 32 位主键。
3. 立即调用 `manual_source_creators({ requirement_id: data.id, size: "20" })`。
4. 只有响应包含非空达人列表，才把它当作后续排序的证据。

不能把数字 `demand_id` 当成工具 ID，也不能拿上周会话里的 ID 重新调用。若已开始搜索，则必须先走完 MCN 分支、字段选择、发送和发送后同步，不能中途跳去人工拓展。

## 当前不能宣称完成的部分

以下不是“待优化”，而是上线前必须拿到证据的边界：

1. **远端输入兼容。** `create_submission_batch` 和 `get_workflow_state` 与批准契约有硬差异；当前检查器为 FAIL，插件应继续阻断。
2. **稳定成功出参。** 当前远端 15 个工具没有发布 `outputSchema`。因此不能从工具名或普通成功文本推导字段存在。
3. **写入幂等与未知结果恢复。** 响应丢失后不能盲重试；需由服务端用持久化键和可查询状态对账。
4. **MCN 询价事实。** 选中机构、去重覆盖、发送、同步、回收必须能由同一业务事实链验证，而不是由本地会话记录拼接。
5. **业务算法。** 硬筛、排序因素、缺失值、权重、版本和解释需要批准并用样本复验。
6. **真实端到端验收。** 需要隔离测试需求、供应商和外发通道，且保留输入、版本、原始响应、执行人、时间和结果。

## 什么证据才能把一项需求标为“已验收”

至少同时具备：

1. 对应 Spec 与 Schema 已批准。
2. 仓库实现和针对该行为的自动化测试通过。
3. 实际部署的 Provider 输入/输出与 Spec 一致，并有当日检查记录。
4. 数据库约束、幂等和恢复在真实环境或等价隔离环境中可复现。
5. 用脱敏真实样本完成一次端到端操作；外发行为必须在安全测试通道内。

例如，“`tools/list` 里有 `create_with_distributions`”只满足接口可见；“本地弹窗已经确认”只满足本地放行条件。只有返回逐供应商 `sent` 明细、可对账写入，并完成后续恢复验证，才可把一次发送链路标为验收通过。
