远程环境已经把 Qdrant 向量召回接进了 `search_creators` 的后端流程，但目前更像“已接线的开发版”，还不能说已经稳定上线：没有本次实测成功记录，二次排序没有真正启用，同步后的对账和失效数据清理也还没补齐。

## 现在实际有什么

### 1. 远程 MCP 仓库只是入口

`D:\yp_local_mcp` 负责对外提供 `search_creators` 等业务工具，再把请求转给达人资料后端。这个仓库本身没有 Qdrant、DashVector、Embedding 或 Rerank 的实现，也没有单独公开下面这些向量工具：

- `search_creator_tag_vectors`
- `health_check_vector_store`
- `sync_creator_tag_vectors`

这与当前正式 Spec 的方向基本一致：用户只需要使用“找达人”和“排达人”，不需要直接操作向量数据库。

### 2. 真正的向量代码在达人资料后端

`D:\data_creator_profiles_api` 在 2026-07-16 的提交 `3b2b9eb` 中加入了达人向量召回，当前使用：

- Qdrant 保存和查询向量；
- 阿里云百炼 `text-embedding-v4` 把文字转成向量；
- `content` 和 `commercial` 两类向量；
- 小红书、抖音达人账号数据；
- 固定达人身份生成方式，重复同步不会因为随机 ID 产生多份记录。

服务器存在被 Git 忽略的 `.env.vector`，本次只确认所需配置项均已填写，没有读取具体值。这只能证明“配置已准备”，不能证明 Qdrant 中已有完整数据，也不能证明线上查询已经成功。

### 3. 找达人时的实际流程

目前 `search_creators` 会先尝试向量召回：

```text
读取需求中的内容、品类和产品信息
→ 隐去需求中的链接、邮箱和手机号
→ 百炼生成向量
→ Qdrant 从 content / commercial 两路找达人 ID
→ 回到 MySQL 读取当前达人和供给关系
→ 用 MySQL 做价格、平台等业务判断
→ 返回候选达人
```

如果向量配置缺失或外部服务失败，流程会继续走普通数据库查询，并在结果里带上 `vector_recall_degraded`。因此向量服务故障一般不会直接阻断找达人。

### 4. 已有人工同步脚本

后端有 `scripts/sync_creator_vectors.py`，可以分批读取小红书或抖音达人数据，生成两类向量并写入 Qdrant。它已经做到：

- 按“更新时间 + 达人 ID”继续上次进度，减少漏数和重复；
- 一整批写入成功后才推进进度；
- 同一个达人使用稳定 ID，重复执行可以覆盖更新；
- 支持只读取、不写入的 dry-run 模式。

但它仍是人工命令，不是完整的日常同步系统。当前没有看到自动调度、进度持久保存、MySQL 与 Qdrant 数量对账、下线达人清理或失败批次管理。

### 5. 二次排序只是“代码已备好”

后端已经写了 `qwen3-rerank` 的调用代码和单元测试，但 `search_creators` 当前没有真正调用它。Qdrant 返回的相似度顺序也没有完整保留到最终结果，所以现在主要作用是缩小候选范围，还不能理解为“已经按内容匹配度排好名次”。

- 当前查询速度、成功率和费用；

- 服务重启后实际运行进程是否加载了 `.env.vector`。

所以当前合适的说法是“代码已接入、配置已准备”，不是“向量数据库已经验收上线”。

## 主要风险

1. **路线反复**：本项目最新设计倾向 DashVector，远程已经投入 Qdrant。如果不先定方向，后续同步、监控和运维代码可能要重写。
2. **可能少返回达人**：向量召回成功但命中的达人没有供给关系时，当前流程可能直接返回空结果，不一定自动回到更宽的普通查询。
3. **相关性顺序会丢失**：向量库找到的先后顺序没有完整传到最终列表，用户看到的顺序未必代表内容更匹配。
4. **旧数据可能残留**：同步脚本能新增和覆盖，但还没有清理已下线或已失效达人，也没有自动对账。
5. **脱敏还不完整**：需求查询会隐去链接、邮箱和手机号；同步达人数据时会直接使用简介和标签，没有看到同等级的清洗步骤。需要确认这些内容是否允许发送给模型服务。
6. **参数尚未业务验收**：模型、维度和召回规则已经写进开发代码，但正式 Spec 仍要求先用固定样本评估，暂时不应当作生产默认规则。
7. **远程正在并行改数据库字段**：两个远程仓库检查前就已有未提交的字段对齐改动。后续修改向量流程时必须基于这些新字段继续，不能覆盖或绕开现有工作。

## 下一步计划

### 第一步：先定路线，不继续扩写两套实现

负责人需要在 Qdrant 和 DashVector 中选一个近期目标。保守建议是：先把已经接入的 Qdrant 当作开发环境的影子能力完成小规模验证，再根据维护成本、费用和效果决定是否换 DashVector。没有评估结果前，不做面向用户的正式切换。

### 第二步：做一次可回退的小规模验收

1. 准备 20～50 条有代表性的真实需求和期望达人。
2. 只同步一小批脱敏达人数据。
3. 对比“普通数据库查询”和“向量辅助查询”的结果、耗时与费用。
4. 验证向量服务断开后仍能正常找达人。
5. 记录小红书、抖音数据量以及与 MySQL 的差异。

### 第三步：补齐上线前的基本保障

- 增加仅供内部使用的健康检查，不向普通用户暴露密钥或连接信息；
- 保存每次同步进度和结果，支持失败后继续；
- 增加失效达人清理和 MySQL/Qdrant 对账；
- 在查询结果中统一说明本次是否用了向量、为什么降级、数据来自哪里；
- 保留向量召回顺序，再决定是否接入二次排序；
- 对达人简介和标签增加明确的清洗与发送范围检查；
- 新版本使用新的 Collection，验证通过后再切换，旧版本短期保留用于回退。

### 第四步：再决定是否正式启用

只有在召回效果、数据一致性、降级、隐私和费用都通过后，才把向量能力从开发/影子状态切成正式能力。即使正式启用，对用户仍只保留 `search_creators` 和 `rank_creators`，不新增需要用户理解的向量工具。
