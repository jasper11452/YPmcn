# 达人召回优化需求

> 创建时间：2026-07-09
> 状态：draft
> 关联表：`xhs_creator_accounts`、`dy_creator_accounts`、`creator_candidate_pool`

## 背景

当前达人召回在多个维度存在过度过滤，导致大量潜在匹配达人被排除。需要从源池规模、匹配策略、硬筛逻辑三个方向做系统性优化，目标：**召回率从当前 ~50% 提升至 80%+**。

### 现状数据

| 维度 | 数值 | 问题 |
|------|------|------|
| 源表规模 | 小红书 500 / 抖音 500 | 达人池过小，无法覆盖长尾需求 |
| 硬筛通过率 | 83.9%（2868/3416） | 16% 被硬筛掉，412 个因预算超限 |
| 内容匹配率 | 48.4%（3316 有分 / 总3416） | 1580 个 account_found=false，达人找不到 |
| 匹配分分布 | 0 分 1580 人，40 分 1696 人，50 分 40 人 | 严重扁平化，无区分度 |
| 标签匹配 | 品类不匹配是主要阻断原因 | 美妆达人 → 母婴需求完全堵死 |

### 核心瓶颈

1. **品类硬匹配**：当前 `category_requirements` 虽标记为 `soft`，但 `expected: ['母婴,亲子']` 与 `actual: ['美妆护肤', ...]` 完全不重叠时仍然直接归零
2. **预算一刀切**：价格超预算直接硬筛失败（548 个失败中 424 个为预算问题，占 77%）
3. **源表太小**：500 个达人库无法支撑多样化需求
4. **分数扁平**：几乎所有通过的都是 40 分，无法做差异化排序

---

## 需求目标

### 一级目标：扩大有效召回池

- **召回覆盖率**：单个需求至少覆盖源池 60% 以上的达人进入候选池（含硬筛前）
- **硬筛通过率**：≥ 90%
- **内容匹配分 > 0 比例**：≥ 85%

### 二级目标：提升匹配质量区分度

- **内容匹配分标准差**：≥ 10（当前几乎全 40 分，标准差 ≈ 0）
- **多维度评分可见**：每个达人展示分类得分、互动得分、性价比得分

---

## 方案设计

### P0：品类匹配由硬匹配改为多级模糊匹配

**现状**：`category_requirements.mode = 'soft'` 但实际行为仍为硬匹配——品类完全不重叠时直接丢弃。

**改造**：品类匹配改为三级降级链：

```
Level 1: 品类完全命中 → score = 100
Level 2: 品类相邻/相关（按预定义品类别名组）→ score = 60
Level 3: 品类不匹配但内容标签（content_theme_label）有关键词重叠 → score = 30
Level 4: 无任何关联 → 不因为品类单独排除，score = 10，保留到综合排序
```

**品类别名组定义**（第一期最小集，后续可配置化）：

| 主品类 | 别名/相关品类 |
|--------|-------------|
| 母婴亲子 | 亲子、育儿、母婴、宝宝、早教 |
| 美妆护肤 | 化妆、护肤、彩妆、成分党、底妆、通勤妆 |
| 游戏动漫 | 桌游、二次元、游戏攻略、COSPLAY |
| 教育职场 | 学习方法、考证、英语提升、职场效率 |
| 健身运动 | 居家训练、健康饮食、塑形 |
| 萌宠 | 养猫、养狗、宠物、宠物用品、训犬 |
| 旅行户外 | 徒步、露营、城市漫游、周末旅行 |
| 汽车出行 | 试驾、新能源、车品、汽车养护 |
| 科技数码 | 手机测评、效率工具、摄影器材、软件推荐 |
| 美食 | 探店、食谱、食材测评 |
| 家居家装 | 收纳、软装、新房装修 |
| 时尚穿搭 | 穿搭、OOTD、服装测评 |

### P1：预算超限由硬筛改为软排序

**现状**：`price_cents > budget_max_cents` → 直接 `hard_filter_failed_reason: PRICE_OVER_BUDGET`，完全排除。

**改造**：

1. 预算超限不再作为硬筛条件，改为「性价比降权」因子
2. 价格分计算规则：

```
超预算 ≤ 20%：price_score = 70
超预算 20%-50%：price_score = 40
超预算 50%-100%：price_score = 10
超预算 > 100%：price_score = 0（但不排除，标记 risk_notes = "报价严重超预算"）
在预算内：price_score = 100
```

3. 无报价（`offer_id IS NULL`）的达人依然排除

### P2：扩大源表达人池

**现状**：`xhs_creator_accounts` 和 `dy_creator_accounts` 各 500 行。

**改造**：
- 短期（本次）：将 `date IS NOT NULL` 的过滤条件去掉，允许旧数据的达人进入候选，但标记 `data_freshness_days = DATEDIFF(NOW(), date)`，30 天以上数据在排序中做衰减
- 中期：扩展数据采集源，增加达人拓展达人入库通道

### P3：多维度评分体系

**现状**：`content_match_score` 全集中在 40 分，无区分度。

**改造**：将 `content_match_score` 拆分为可解释的多维评分：

```json
{
  "content_match_score": 0-100（综合加权分）,
  "score_detail": {
    "category_score": 0-100,       // 品类匹配（P0 多级模糊）
    "keyword_score": 0-100,        // 内容标签关键词重叠度
    "audience_score": 0-100,       // 受众匹配（性别/年龄/城市分布与需求对比）
    "engagement_score": 0-100,     // 互动质量（互动率、完播率、爆文率加权）
    "price_value_score": 0-100,    // 性价比（P1 预算弹性 + CPM/CPE对比）
    "freshness_score": 0-100,      // 数据新鲜度（P2 date字段）
    "risk_penalty": -50-0          // 风险扣分（档期不确定、报价异常等）
  },
  "weight_config": {
    "category": 0.25,
    "keyword": 0.20,
    "audience": 0.15,
    "engagement": 0.15,
    "price_value": 0.15,
    "freshness": 0.10
  }
}
```

### P4：标签关键词重叠度从精确匹配改为 Jaccard 相似度

**现状**：`content_theme_label`、`content_type_label` 等标签字段做精确匹配，导致大量语义相近但字面不同的标签被排除。

**改造**：

```
keyword_score = Jaccard(需求关键词集合, 达人标签集合) × 100

需求关键词集合 = 从 raw_messages_json 提取的需求侧标签 + 品牌词 + 产品词
达人标签集合 = content_theme_label + industry_tag_label + talent_type_label 拆分去重
```

## 验收标准

### 功能验证

1. 同一个需求（如「母婴亲子」品类需求），召回结果应包含非「母婴亲子」品类达人（例如萌宠、美妆等相邻品类），并在 `category_score` 上体现降权
2. 预算超限 ≤ 50% 的达人应出现在候选池中，`price_value_score` < 100
3. `score_detail` 包含全部 6 个维度分值
4. `content_match_score` 的 stddev > 10（抽样 10 个需求验证）
5. 无报价达人（`offer_id IS NULL`）不进入最终推荐列表

### 回归验证

- `hard_filter_passed = 0` 仅保留真正不可用的达人（如无报价、被封禁等）
- 已有测试用例 `tests/goldens/requirement_cases.json` 全部通过

## 风险与注意事项

1. **召回过多反而干扰决策**：放宽后单个需求可能召回 200+ 达人，需在排序层做好截断（top-N 取前 50 或前 100）
2. **品类别名维护**：P0 的品类别名组需要业务方确认和持续维护
3. **价格边界**：超预算 100% 以上的达人标记风险但不排除，前端需有价格预警展示
4. **旧数据质量**：P2 放开 date 限制后，需确认 30 天以上数据仍有参考价值
