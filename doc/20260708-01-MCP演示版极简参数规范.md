# MCP MVP 极简参数规范（字段表对齐版）

> 目标：按 `/Users/jasper/Desktop/小红书_抖音字段合并表.csv` 建立统一字段契约，让需求解析表、需求表、达人表、后续 MCP 工具全部对齐。  
> 结论：工具保留，但后续工具只传上一步生成的业务 `id`。真正要重做的是 `validate_requirement` 的字段映射和校验规则。

## 1. 核心结论

MVP 的最小链路：

```text
validate_requirement
  -> search_creators
  -> rank_mcns
  -> 媒介确认供给比例和 MCN 清单
  -> 并行：
       create_with_distributions -> 等待 MCN 回填 -> ingest_mcn_submissions
       manual_source_creators -> 手扒达人入池
  -> 汇总 MCN 回填达人 + 手扒达人 + 刊例候选
  -> rank_creators
  -> create_submission_batch
```

所有下游工具都不再接收一堆筛选参数，而是只接收数据库中上一阶段生成的 `id`。

| 阶段 | 工具 | 最小入参 | 最小输出 |
|---|---|---|---|
| 需求解析和入库 | `validate_requirement` | Agent 解析后的需求 JSON | `id` = 需求表 32 位主键 |
| 搜索达人 | `search_creators` | `id` = 需求表主键 | `id` = 候选池/搜索结果 ID，外加候选摘要 |
| MCN 排序 | `rank_mcns` | `id` = 候选池/搜索结果 ID | `id` = MCN 排序方案 ID |
| 企微发送 | `create_with_distributions` | `id` = MCN 排序方案 ID + Hook 必需发送字段 | `id` = 分发/询价 ID |
| 回填导入 | `ingest_mcn_submissions` | `id` = 分发/询价 ID | `id` = 回填批次 ID |
| 手扒导入 | `manual_source_creators` | `id` = 需求表主键 + 手扒来源 | `id` = 手扒批次 ID |
| 达人精排 | `rank_creators` | `id` = 需求表主键 | `id` = 精排运行 ID |
| 最终提报 | `create_submission_batch` | `id` = 精排运行 ID | `id` = 提报批次 ID + 提报表 |

> 重要修正：这里的 `id` 不是旧文档里的 `demand_id + demand_version`，而是需求表里 `id(32位uuid)` 主键。`demand_id` / `demand_version` 可以保留为内部版本字段，但不再作为 MCP 下游参数。

## 2. 字段表是唯一契约

字段合并表分两类字段：

| 字段区域 | 用途 |
|---|---|
| 上半部分达人字段，例如 `platform`、`followercount`、`organization`、`content_tag`、`download_price_l1` | 客户额外筛选条件要映射到这些字段，后续 `search_creators` 用同字段跟达人表匹配 |
| 底部项目/需求字段，例如 `id`、`submission_deadline_at`、`budget_max_cents`、`quantity_total` | 需求表自身字段，用于流程、预算、截止时间、数量等 |

MVP 不允许 Agent 自己创造字段名。所有可执行字段必须来自字段合并表。

如果客户说“粉丝量大于等于一万”，Agent 应映射为：

```json
{
  "field": "followercount",
  "operator": ">=",
  "value": 10000
}
```

如果客户说“要妈妈类达人”，但无法确定应该映射到 `content_tag`、`talent_type_label`、`pgy_blogger_type_label` 还是其他字段，必须向媒介确认，不能擅自落字段。

## 3. `validate_requirement` 的真实职责

`validate_requirement` 不是简单存 JSON。它必须完成 4 件事：

1. 校验需求表必填字段是否非空。
2. 把客户额外需求拆成字段合并表中的确定字段。
3. 对语义模糊字段返回澄清问题，等待媒介确认。
4. 所有字段都确定后，创建一张需求表记录，生成 32 位主键 `id`。

### 3.1 必填字段校验

按当前字段合并表，底部标记为“必填”的字段是：

| 字段 | 含义 |
|---|---|
| `id` | 主键 ID，由系统生成 |
| `demand_id` | 需求 ID，可内部生成 |
| `demand_version` | 需求版本，可内部生成 |
| `submission_deadline_at` | 提交时间 |
| `submission_deadline_raw` | 提交时间原文 |
| `raw_messages_json` | 原始输入需求 JSON |
| `budget_min_cents` | 预算下限 |
| `budget_max_cents` | 预算上限 |
| `budget_raw` | 预算原文 |
| `rebate_min_rate` | 返点下限 |
| `rebate_max_rate` | 返点上限 |
| `rebate_raw` | 返点原文 |
| `quantity_total` | 需要数量 |
| `status` | 状态 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

执行规则：

| 类型 | 规则 |
|---|---|
| 系统生成字段 | `id`、`demand_id`、`demand_version`、`status`、`created_at`、`updated_at` 由 MCP/DB 生成，不要求媒介填写 |
| 媒介必须补全字段 | `submission_deadline_at/raw`、`raw_messages_json`、预算、返点、`quantity_total` 必须非空 |
| 缺失处理 | 返回 `status=blocked` 或 `status=need_clarification`，列出缺失字段，不能继续 `search_creators` |

> 注意：如果业务上允许返点缺失或预算缺失，需要先改字段合并表的“必填/可空”规则。MCP 不应在字段表标“必填”的情况下自己放宽。

### 3.2 选填字段映射

媒介可能已经填完必填项，但还会有额外需求，例如：

| 客户表达 | 字段映射 | 执行 |
|---|---|---|
| 粉丝量大于等于一万 | `followercount >= 10000` | 可直接落库 |
| 小红书平台 | `platform = xhs` | 可直接落库 |
| 近 30 天有发文 | `has_social_30day = true` | 可直接落库 |
| 已入驻 MCN | 小红书可映射 `has_organization=true`，通用可参考 `organization not empty` | 如果平台未定，需问 |
| 妈妈类达人 | 可能是 `content_tag` / `talent_type_label` / 平台专属标签 | 语义模糊，必须问 |
| 性价比高 | 可能是价格、CPE、CPM、互动等多个字段 | 必须问 |

建议落库结构：

```json
{
  "id": "32位uuid",
  "raw_agent_json": {},
  "required_fields": {},
  "field_filters": [
    {
      "field": "followercount",
      "operator": ">=",
      "value": 10000,
      "source_text": "粉丝量大于等于一万",
      "confidence": "high"
    }
  ],
  "ambiguous_fields": []
}
```

`field_filters` 中的 `field` 必须能在字段合并表里找到。`confidence=medium/low` 或多字段候选时，不允许落成最终可执行筛选条件，必须向媒介询问。

### 3.3 模糊字段必须问

`validate_requirement` 遇到以下情况必须停：

| 情况 | 处理 |
|---|---|
| 一个表达可能对应多个字段 | 返回候选字段和问题 |
| 字段存在平台差异，且需求没有明确平台 | 询问平台或字段口径 |
| 数值单位不清楚 | 询问单位 |
| 预算/返点/提交时间等必填项缺失 | 询问补全 |
| 客户用了主观词，如“优质”“性价比高”“垂类强” | 询问具体指标 |

澄清响应示例：

```json
{
  "success": true,
  "data": {
    "status": "need_clarification",
    "missing_required_fields": [],
    "ambiguous_fields": [
      {
        "source_text": "妈妈类达人",
        "candidate_fields": ["content_tag", "talent_type_label", "pgy_blogger_type_label"],
        "question": "这里的“妈妈类达人”是按内容类目筛，还是按达人人设/博主类型筛？"
      }
    ]
  },
  "error": null,
  "trace_id": "trace_xxx"
}
```

只有当必填字段完整且所有可执行字段都已确认后，才能返回：

```json
{
  "success": true,
  "data": {
    "id": "6f1b7b2f0f3b4e0a9e7b6d8c1a2b3c4d"
  },
  "error": null,
  "trace_id": "trace_xxx"
}
```

## 4. `search_creators` 的真实职责

`search_creators` 只传需求表主键 `id`：

```json
{
  "id": "6f1b7b2f0f3b4e0a9e7b6d8c1a2b3c4d"
}
```

执行逻辑：

1. 读取需求表记录。
2. 提取所有非空且已确认的字段。
3. 将这些字段与达人表同名字段进行匹配。
4. 数值字段按操作符精准筛选。
5. 布尔/枚举/文本字段按字段规则匹配。
6. 内容类自然语言字段先不做复杂语义优化，MVP 可只做标签/字段包含匹配。
7. 写入候选池。

示例：

```json
{
  "field_filters": [
    {"field": "platform", "operator": "=", "value": "xhs"},
    {"field": "followercount", "operator": ">=", "value": 10000},
    {"field": "budget_max_cents", "operator": ">=", "value": "达人报价字段"}
  ]
}
```

候选池最小写入：

| 字段 | 来源 |
|---|---|
| `requirement_id` | 需求表 `id` |
| `platform` | 达人表 |
| `platform_account_id` / `kw_uid` | 达人表 |
| `organization` / `mcn_id` | 达人表或 MCN 映射表 |
| `matched_fields_json` | 命中的需求字段 |
| `unmatched_fields_json` | 未命中的需求字段 |
| `candidate_source` | `rate_card` |

`search_creators` 输出：

```json
{
  "success": true,
  "data": {
    "id": "6f1b7b2f0f3b4e0a9e7b6d8c1a2b3c4d",
    "summary": {
      "candidate_count": 120,
      "mcn_candidate_count": 86,
      "non_mcn_candidate_count": 34
    }
  },
  "error": null,
  "trace_id": "trace_xxx"
}
```

## 5. `rank_mcns` 的真实职责

`rank_mcns` 入参只传 `search_creators.data.id`：

```json
{
  "id": "candidate_pool_123"
}
```

执行逻辑：

1. 读取这批候选池/搜索结果中的达人。
2. 提取达人所属 MCN：优先 `mcn_id`，其次 `organization`。
3. 按 MCN 聚合达人数量、报价、返点、历史评级等。
4. 输出 MCN 排序清单。
5. 计算供给比例：MCN 达人、未签约/可手扒达人、需求数量。
6. 判断是否需要手扒补量。

输出必须包含：

```json
{
  "success": true,
  "data": {
    "id": "mcn_plan_123",
    "candidate_pool_id": "candidate_pool_123",
    "requirement_id": "6f1b7b2f0f3b4e0a9e7b6d8c1a2b3c4d",
    "source_mix_summary": {
      "required_creator_count": 20,
      "matched_creator_count": 120,
      "mcn_creator_count": 86,
      "non_mcn_creator_count": 34,
      "mcn_ratio": 0.7167,
      "non_mcn_ratio": 0.2833,
      "manual_sourcing_recommended": true,
      "manual_sourcing_reason": "非 MCN 达人占比 28.3%，建议同步手扒补充"
    },
    "mcns": []
  },
  "error": null,
  "trace_id": "trace_xxx"
}
```

媒介确认后可以并行推进：

```text
create_with_distributions 发送 MCN 企微
manual_source_creators 启动/导入手扒达人
```

## 6. 企微发送和等待回填

`create_with_distributions` 保留现有 Hook 必需字段：

```json
{
  "id": "mcn_plan_123",
  "deadline": "2026-07-10T18:00:00+08:00",
  "remindAt": "2026-07-10T12:00:00+08:00",
  "usageScope": "project",
  "supplierIds": ["supplier_001"],
  "preview_only": false
}
```

如果后端能从 `mcn_plan_123` 反查 `deadline/remindAt/supplierIds`，未来可以继续压缩；但当前 Hook 会校验这些字段，MVP 不建议省。

正式发送后：

1. 系统进入等待回填。
2. 同步进行手扒。
3. 设置定时任务，到时间后触发回填汇总。

> 注意：之前 Hook 文档说“当前不创建 Cron”。如果现在业务要求设置定时任务，需要补一个明确的调度机制：可以是后端任务，不一定是 Agent 自动 Cron，但必须有可追踪的 `scheduled_job_id`。

## 7. `ingest_mcn_submissions` 和 `manual_source_creators`

### 7.1 MCN 回填导入

最小入参：

```json
{
  "id": "distribution_123"
}
```

如果无法通过分发 ID 自动读取回填表，则加：

```json
{
  "id": "distribution_123",
  "source_ref": "file_or_sheet_456"
}
```

必须写入：

| 表 | 内容 |
|---|---|
| `mcn_submission_items` | MCN 回填原始明细 |
| `creator_supply_offers` | 报价/返点/档期 |
| `creator_candidate_pool` | `candidate_source=mcn_returned` |

### 7.2 手扒达人导入

最小入参：

```json
{
  "id": "6f1b7b2f0f3b4e0a9e7b6d8c1a2b3c4d",
  "source_ref": "manual_sheet_123"
}
```

或直接传：

```json
{
  "id": "6f1b7b2f0f3b4e0a9e7b6d8c1a2b3c4d",
  "items": [
    {
      "platform": "xhs",
      "platform_account_id": "xxx"
    }
  ]
}
```

必须写入候选池：

```text
candidate_source = manual_search
```

## 8. `rank_creators` 精排

`rank_creators` 入参只传需求表主键：

```json
{
  "id": "6f1b7b2f0f3b4e0a9e7b6d8c1a2b3c4d"
}
```

执行逻辑：

1. 读取该需求下所有候选：
   - `rate_card`
   - `mcn_returned`
   - `manual_search`
2. 去重同一达人。
3. 再次按需求表非空字段复筛。
4. 按报价、返点、匹配字段数量、数据表现等排序。
5. 写入精排运行和推荐达人表。

输出：

```json
{
  "success": true,
  "data": {
    "id": "run_123",
    "summary": {
      "ranked_count": 40,
      "mcn_returned_count": 25,
      "manual_count": 8,
      "rate_card_count": 7
    }
  },
  "error": null,
  "trace_id": "trace_xxx"
}
```

## 10. 需要同步修改的地方

因为现在第一步的契约变了，必须同步改：

| 模块 | 要改什么 |
|---|---|
| MCP `validate_requirement.inputSchema` | 支持 Agent 解析后的结构化需求 JSON |
| Hook `before_tool_call` | 不再只允许旧 4 个字段，或明确把结构化 JSON 放进允许字段 |
| `check_requirement_params.py` | 按字段合并表校验必填、类型、单位 |
| Skill `requirement-intake.md` | 改成“Agent 先按字段表解析，模糊必问，确认后调用 MCP” |
| Skill `requirement-parsing.md` | 补字段映射、模糊字段询问规则 |
| 数据库需求表 | 支持 32 位主键 `id`，并保存 `field_filters` / `ambiguous_fields` / `raw_agent_json` |
| `search_creators` | 从需求表取非空字段，与达人表同字段匹配 |

## 11. MVP 成功标准

| 阶段 | 可验证结果 |
|---|---|
| 必填校验 | 缺必填字段时不生成需求 `id`，返回补全问题 |
| 字段映射 | 额外需求都落到字段合并表字段；模糊字段会问媒介 |
| 需求入库 | 生成 32 位主键 `id` |
| 搜索达人 | `search_creators(id)` 从需求表读取非空字段并筛选达人 |
| MCN 排序 | `rank_mcns(id)` 输出机构排序和 MCN/非 MCN/手扒建议比例 |
| 企微发送 | 发送 MCN 询价并进入等待回填 |
| 回填 + 手扒 | 回填达人和手扒达人都进入候选池 |
| 精排 | `rank_creators(id)` 对汇总后的达人排序 |
| 提报 | `create_submission_batch(run_id)` 输出最终提报表 |
