# YPmcn MVP 完整链路总览

> 更新日期：2026-07-09  
> 适用范围：当前 `YPmcn` Skill + Hooks + MCP 演示链路。  
> 目标：先跑通完整流程，保留关键停顿和提醒，不用生产版复杂表结构、账本和 trace 细项阻断主流程。

## 1. 结论

当前 MVP 是一条“数据库状态对象 + MCP 工具 + Skill 引导 + Hooks 门控”的链路。

```text
客户 Brief
  -> Agent 按字段表解析为 validate_requirement JSON，必填项齐全后继续
  -> Agent 将额外需求匹配到 creator_candidate_pool_schema.csv 字段并让用户确认
  -> validate_requirement 写 customer_demands，返回 data.id（customer_demands 主键），demand_id/demand_version 只作版本字段
  -> search_creators 用 validate_requirement.data.id 读需求表并匹配 xhs_creator_accounts / dy_creator_accounts，写 creator_candidate_pool
  -> rank_mcns 用 search_creators.data.id 聚合 MCN，计算供需关系和询价建议，写 mcn_recommendation_items
  -> 前端展示供需关系、建议手扒比例和建议询价 MCN 列表
  -> 用户同意或修改 MCN 列表
  -> Agent 根据 validate_requirement.data.id 读取需求表非空字段，拟写企微消息让用户确认
  -> create_with_distributions 用固定企微接口字段创建项目分发/企微询价，为每个 MCN 生成唯一填报链接并预填该 MCN 候选达人
  -> 需要手扒时同步启动手扒程序
  -> 停，等 MCN 回填和手扒结果
  -> ingest_mcn_submissions / manual_source_creators 把 MCN 回填和手扒结果写回候选
  -> rank_creators 汇总候选，统一去重、筛选、精排，写 recommendation_runs / creator_recommendation_items
  -> create_submission_batch 用 run_id 生成首批提报表给媒介看
  -> record_client_feedback 记录客户反馈并决定补批、重排或回到需求
```

MVP 的关键不是把每个细节都校验到极致，而是保证：

1. 每一步都有数据库主键可恢复。
2. 每个下游工具只吃上一步成功结果里的业务 ID。
3. 该停的地方必须停：额外需求字段确认、结构化 Brief 确认、MCN 询价名单确认、企微消息确认、分发后精排确认、风险账号确认。
4. Hooks 只拦流程顺序、发送权限、未来时间、`usageScope`、必要链路 ID 和等待锁。
5. Hooks 不因为 `trace_id`、响应信封细项或语义启发式疑点改写/阻断结果。

## 2. 分层职责

| 层 | 当前职责 | 不做什么 |
|---|---|---|
| Agent | 读 Skill、按 schema 组织参数、把 Brief 解析成 JSON、在人工节点弹窗确认、展示结果摘要 | 不模拟 MCP 成功、不绕过工具、不自己写库 |
| Skill | 告诉 Agent 按什么顺序调工具、哪些节点必须停、怎么回复媒介 | 不承载真实业务状态 |
| Hooks | 调用前门控、分发后等待锁、状态缓存、阻断 Bash/curl 直连企微接口 | 不做复杂业务语义校验、不要求 trace 细项 |
| MCP 工具 | 读写业务表、生成每一步业务 ID、做筛选/排序/分发/提报 | 不依赖 Agent 传 schema 外重复上下文 |
| 数据库 | 保存需求、候选、MCN 询价建议、回填、精排、提报、反馈，可恢复流程 | MVP 不改旧表名，不做完整生产级审计账本 |

## 3. 版本和主键流

表名沿用旧设计；运行时按当前 schema 使用上一步 `data.id` 串联。这个 `id` 映射旧业务表主键，不代表新增表名；`demand_id + demand_version` 保留为 `customer_demands` 内部版本事实，不再作为下游主入参。

| 阶段 | 成功输出 | 下一步怎么用 |
|---|---|---|
| `validate_requirement` | `data.id`（`customer_demands` 主键） | `search_creators({ id: validate_requirement.data.id, ... })` |
| `search_creators` | `data.id`、`candidate_pool_written=true`、候选写入 `creator_candidate_pool` | `rank_mcns({ id: search_creators.data.id, ... })` |
| `rank_mcns` | `data.id`、`inquiry_advice`、`mcn_recommendation_items` | 媒介确认 MCN 后 `create_with_distributions({ id: rank_mcns.data.id, ...固定企微字段 })` |
| `create_with_distributions` | 项目/分发结果、每个供应商 token，必要时关联 `mcn_inquiries` | `ingest_mcn_submissions({ inquiry_id, items })` |
| `rank_creators` | `run_id` | `create_submission_batch({ run_id })` |
| `create_submission_batch` | `batch_no` / `submissions[]` | `record_client_feedback({ run_id, feedback_items })` |

## 4. 端到端阶段

### 4.1 需求进入

客户给出自然语言 Brief。Agent 做第一层解析，把可确定字段转成 JSON。

当前必填项：

| 字段 | 说明 |
|---|---|
| `platform` | 平台，`xhs` 或 `dy` |
| `submission_deadline_at` | 提报截止时间 |
| `raw_messages_json` | 客户原始需求 JSON |
| `budget_min_cents` | 单价/预算区间下限，单位分 |
| `budget_max_cents` | 单价/预算区间上限，单位分 |
| `budget_raw` | 单价/预算原文 |
| `rebate_min_rate` | 返点区间下限，小数 |
| `rebate_max_rate` | 返点区间上限，小数 |
| `rebate_raw` | 返点原文 |
| `quantity_total` | 需要达人数量 |

单价/预算和返点都必须做成区间。原文只给单值时写成闭区间，例如 `返点20%` -> `rebate_min_rate=0.2`、`rebate_max_rate=0.2`。原文只给上限时，下限按业务可接受下界归一，例如 `预算3万以内` -> `budget_min_cents=0`、`budget_max_cents=3000000`。

示例：

```json
{
  "platform": "xhs",
  "quantity_total": 10,
  "submission_deadline_at": "2026-07-10T18:00:00+08:00",
  "raw_messages_json": "[{\"role\":\"client\",\"content\":\"小红书找10个美妆达人，预算3万以内，返点20%，明天18点前提报\"}]",
  "budget_min_cents": 0,
  "budget_max_cents": 3000000,
  "budget_raw": "3万以内",
  "rebate_min_rate": 0.2,
  "rebate_max_rate": 0.2,
  "rebate_raw": "返点20%",
  "raw_messages": [
    {"role": "client", "content": "小红书找10个美妆达人，预算3万以内，返点20%，明天18点前提报"}
  ]
}
```

缺任一必填项时不可继续。`validate_requirement` 写入的需求主表仍是 `customer_demands`，字段口径以 `YPmcn/skills/media-assistant/references/creator_candidate_pool_schema.csv` 的 `字段` 列为准；其中追加的需求字段就是需求主表表头：`id(32位uuid)`、`demand_id`、`demand_version`、`submission_deadline_at`、`submission_deadline_raw`、`raw_messages_json`、`budget_min_cents`、`budget_max_cents`、`budget_raw`、`rebate_min_rate`、`rebate_max_rate`、`rebate_raw`、`quantity_total`、`status`、`created_at`、`updated_at`、`note`、`project_name`、`brand`、`product`、`project_start_start`、`project_start_end`。

达人资源库不做统一 `creators` 表，按平台拆为 `xhs_creator_accounts` 和 `dy_creator_accounts`。两张平台资源库的字段从同一份 `creator_candidate_pool_schema.csv` 继承：共同字段保持同名，平台专属字段只存在于对应平台表；额外需求只能映射到这份字段表里的可执行字段，不确定时返回澄清问题，不落最终筛选条件。

### 4.2 需求入库

`validate_requirement` 校验字段、写 `customer_demands`，返回 `data.id`。如后端同时返回 `demand_id + demand_version`，只作为需求版本事实保存和展示，不作为下游主入参。

如果缺必填字段，返回 `status=draft` 或类似状态，Agent 停下让媒介补充，不继续 `search_creators`。

如果必填项齐全且额外需求字段映射已确认，Agent 仍要停在 `confirm-structured-brief`，让媒介确认结构化 Brief。确认后才能调用 `search_creators`。

### 4.3 候选搜索

`search_creators` 按运行时 schema 接收 `id`，取值为 `validate_requirement.data.id`，可带 `authorized_relaxations`、`category_hard_filter_confirmation`、`write_candidate_pool`、`limit` 等运行时 schema 允许字段。

MCP 从 `customer_demands` 读取平台、数量、预算、返点、内容要求和筛选条件，再按 `platform` 匹配 `xhs_creator_accounts` 或 `dy_creator_accounts`，写入：

- `creator_candidate_pool`：本轮候选达人池，包含 `candidate_source`、`matched_json`、`content_match_score`、`source_detail_json` 等旧接口字段。

输出 `candidate_pool_written=true` 后，进入 MCN 排序。

### 4.4 MCN 排序和发送确认

`rank_mcns` 按运行时 schema 接收 `id`，取值为 `search_creators.data.id`。

MCP 从 `creator_candidate_pool` 读取候选达人，再提取相关 MCN 在数据库里的内容，生成 `mcn_recommendation_items` 和 `inquiry_advice`。核心输出不是“已发送”，而是供需判断和询价建议。前端展示：

- 当前达人供需关系：需求数量、候选数量、MCN 覆盖数量、缺口。
- 建议手扒比例和原因。
- 建议发送询价的 MCN 列表。
- 表单字段覆盖情况。

用户可以同意或修改 MCN 列表。然后 Agent 根据 `validate_requirement.data.id` 读取 `customer_demands` 非空字段，拟写企微询价消息给用户确认。然后必须依次停：

```text
confirm-supply-ratio
-> mcn-select-for-wechat
-> confirm-form-fields
-> confirm-wecom-permission
-> mcn-wechat-send
```

这些确认完成后才能调用 `create_with_distributions`。如果需要手扒，同步启动手扒程序，不等 MCN 回填后才启动。

### 4.5 企微分发

`create_with_distributions` 使用固定企微发送接口字段，不能为了业务说明随意修改输入输出。调用时不新增工具名，也不恢复旧的 `create_mcn_inquiries`；发送对象来自已确认的 `rank_mcns.inquiry_advice` / `mcn_recommendation_items`，并带上企微接口必需字段：

```json
{
  "projectName": "美妆达人提报",
  "description": "请按表单回填可提报达人、报价、返点和档期。",
  "deadline": "2026-07-10T18:00:00+08:00",
  "remindAt": "2026-07-10T12:00:00+08:00",
  "usageScope": "project",
  "supplierIds": ["supplier_001"],
  "sendWechatNotification": true,
  "prefillRowsBySupplier": {
    "supplier_001": [
      {"talentName": "达人A", "price": 200000}
    ]
  },
  "preview_only": false
}
```

`prefillRowsBySupplier` 使用企微发送接口已经支持的固定字段。每个 MCN 只预填候选池中属于当前 MCN/供应商的达人，作为给机构的推荐底稿。每个 MCN 的唯一填报链接由后端分发响应生成，Agent 不自己拼链接。

当前 Hooks 会检查：

- `deadline/remindAt` 必须是未来的带时区 ISO 时间。
- `supplierIds/supplier_ids` 必须非空。
- `usageScope` 缺失时补为 `project`；显式传错则阻断。
- 只能用 MCP 工具，不能 Bash/curl 直连接口。
- 角色若能识别，必须是媒介或采购。

正式发送成功后 Hooks 设置等待锁。Agent 必须停，等待 MCN 回填和手扒结果；不能因为“消息已发送”直接进入精排。

### 4.6 回填、手扒和精排

MCN 回填通过 `ingest_mcn_submissions` 导入，手扒通过 `manual_source_creators` 导入。两者最终都要回收到候选池，和原始库内候选一起统一处理：

- MCN 回填：`candidate_source = mcn_returned`
- 手扒：`candidate_source = manual_sourced`
- 库内搜索：`candidate_source = rate_card`

`rank_creators` 汇总这些候选，统一去重、复筛、排序，返回 `run_id`。去重和筛选在精排前完成，避免同一达人从 MCN 回填、手扒、库内候选重复进入首批提报。

如果存在风险账号，Agent 停在 `confirm-risky-submission`。用户确认后，`create_submission_batch` 才能带 `allow_need_confirm_with_risk: true`。

### 4.7 提报和反馈

`create_submission_batch` 复用 `rank_creators` 的 `run_id`，先生成首批提报表给媒介看。媒介确认后再进入后续客户提报或反馈流程。

客户反馈后，`record_client_feedback` 记录反馈并给出下一步：

| 反馈类型 | 下一步 |
|---|---|
| 只淘汰部分达人 | 复用当前 `run_id` 继续下一批 |
| 偏好变化但需求核心不变 | 重新 `rank_creators` |
| 需求核心字段变化 | 回到 `validate_requirement` |
| 需要人工处理 | 停止自动流程 |

## 5. 当前状态管理

MVP 不依赖 `get_workflow_state` 工具。状态来自三处：

1. MCP 成功响应里的业务 ID。
2. 可选 `workflow_state` / `allowed_actions`。
3. Hooks 在当前会话里缓存的已完成步骤和等待锁。

最小状态对象：

```json
{
  "phase": "requirement_draft|requirement_ready|candidate_pool_ready|mcn_planning|waiting_mcn_return|candidate_pool_enriched|recommendation_ready|submission_batch_ready|feedback_routing",
  "requirement_id": null,
  "inquiry_ids": [],
  "run_id": null,
  "batch_no": null,
  "project_distribution_completed": false,
  "wait_gate": null
}
```

恢复规则：

1. 有 `run_id`：优先 `get_recommendation_run_detail(run_id)`。
2. 有平台账号：用 `get_creator_detail` 查详情。
3. 只有不明业务 ID：按最近成功响应判断它属于哪个阶段；证据不足就停。
4. 写调用超时/断连：不盲目重试，用业务查询或 `trace_id` 交后端排查。
