# YPmcn MVP 工具契约和输入输出

> 更新日期：2026-07-09  
> 原则：`inputSchema` 是运行时唯一权威；本文定义 MVP 期望契约和 ID 流，不替代线上 schema。

## 1. 总规则

1. 每次调工具前读取运行时 `inputSchema`，只传 schema 允许字段。
2. 下游按当前运行时 schema 使用上一步 `data.id` 串联；这个 `id` 映射旧业务表主键，不代表新增表名。`demand_id + demand_version` 保留为 `customer_demands` 里的版本字段。
3. 工具名、表名和历史设计保持一致，不把 `requirements/candidate_pools/candidate_items/mcn_plans/rank_runs` 作为新的契约表名。
4. 缺 `trace_id`、响应信封细项不阻断主流程。
5. 写调用超时/断连后不盲目重试，用业务 ID 查询或交后端排查。
6. 只有 `success=true` 且存在业务 ID/业务结果时，Agent 才能说“已完成”。
7. 企微发送接口输入输出字段固定，Agent 不发明新字段；预填达人使用接口已支持的 `prefillRowsBySupplier` / `prefill_rows_by_supplier` 等字段。

## 2. 工具总览

| 顺序 | 工具 | 最小输入 | 关键输出 | 下一步 |
|---|---|---|---|---|
| 1 | `validate_requirement` | 结构化需求字段或 `raw_messages` | `data.id`、`status`，可带 `demand_id/demand_version` | `confirm-structured-brief` |
| 2 | `search_creators` | `id`（来自 `validate_requirement.data.id`） | `data.id`、`candidate_pool_written`、供给评估 | `rank_mcns` |
| 3 | `rank_mcns` | `id`（来自 `search_creators.data.id`） | `data.id`、`inquiry_advice`、供需关系、建议手扒比例、建议询价 MCN 列表 | 用户确认/修改 MCN 列表 |
| 4 | `create_with_distributions` | `id`（来自 `rank_mcns.data.id`）+ 固定企微发送字段 + 每 MCN 预填达人 | 每 MCN 唯一填报链接/分发 ID/询价 ID | 等回填和手扒 |
| 5 | `ingest_mcn_submissions` | `inquiry_id` + 真实回填 `items` | 回填导入结果 | 候选汇总 |
| 6 | `manual_source_creators` | `id`（当前需求/候选上下文）+ 手扒结果 | 手扒导入结果 | 候选汇总 |
| 7 | `rank_creators` | `id`（汇总后的候选/回填上下文）+ `ranking_strategy` | `run_id`，统一去重筛选后的精排结果 | 风险确认/首批提报 |
| 8 | `create_submission_batch` | `run_id` | `batch_no` / `submissions[]`，首批提报表 | 媒介查看/客户反馈 |
| 9 | `record_client_feedback` | `run_id` + `feedback_items` | `next_action` | 补批/重排/需求变更 |

只读/审计工具：

| 工具 | 输入 | 用途 |
|---|---|---|
| `get_workflow_state` | 按运行时 schema，常见为业务 `id` 或 `trace_id` | 兼容只读查询；当前 Skill 不强依赖 |
| `get_recommendation_run_detail` | `run_id` | 恢复精排、提报、回填关联 |
| `get_creator_detail` | 按运行时 schema，常见为 `creator_id` 或 `platform + platform_account_id` | 查询达人详情 |
| `audit_manual_adjustment` | `run_id`, `adjustments`, `operator_id` | 记录人工删改/替换/重排 |

## 3. `validate_requirement`

### 3.1 作用

把客户 Brief 变成可执行需求记录。

Agent 先把明确字段解析成 JSON，例如：

```json
{
  "platform": "xhs",
  "quantity_total": 10,
  "submission_deadline_at": "2026-07-10T18:00:00+08:00",
  "raw_messages_json": "[{\"role\":\"client\",\"content\":\"小红书找10个美妆达人，预算3万以内，返点20%，明天18点前提报\"}]",
  "budget_min_cents": 0,
  "budget_max_cents": 3000000,
  "budget_raw": "3万以内",
  "rebate_min_rate": 0.2,
  "rebate_max_rate": 0.2,
  "rebate_raw": "返点20%",
  "raw_messages": [
    {"role": "client", "content": "小红书找10个美妆达人，预算3万以内，返点20%，明天18点前提报"}
  ]
}
```

当前必填项：

```text
platform
submission_deadline_at
raw_messages_json
budget_min_cents
budget_max_cents
budget_raw
rebate_min_rate
rebate_max_rate
rebate_raw
quantity_total
```

单价/预算和返点必须做成区间。原文只给单值时写成闭区间；原文只给上限时下限按业务可接受下界归一。缺失任何必填项时不可继续。

`validate_requirement` 写入的需求主表是 `customer_demands`，表头按 `YPmcn/skills/media-assistant/references/creator_candidate_pool_schema.csv` 的 `字段` 列追加需求字段执行：

```text
id(32位uuid)
demand_id
demand_version
submission_deadline_at
submission_deadline_raw
raw_messages_json
budget_min_cents
budget_max_cents
budget_raw
rebate_min_rate
rebate_max_rate
rebate_raw
quantity_total
status
created_at
updated_at
note
project_name
brand
product
project_start_start
project_start_end
```

必填项满足后，如果还有额外需求，Agent 参考同一份 CSV 匹配需求字段或达人筛选字段，并向用户确认。能落到 CSV 字段的条件才进入筛选；不确定字段只返回澄清问题，不新增主表字段。

达人资源库按平台拆表，不使用统一 `creators` 表。`xhs_creator_accounts` 继承 CSV 中“小红书是否存在=是”的字段，`dy_creator_accounts` 继承 CSV 中“抖音是否存在=是”的字段；两平台重复字段保持同名，平台专属字段只在对应资源库存在。

### 3.2 数据库动作

读：

- 可选旧需求记录，用于需求修改。

写：

- `customer_demands`

### 3.3 输出

可继续：

```json
{
  "success": true,
  "data": {
    "id": "demand_123",
    "demand_id": "demand_123",
    "demand_version": 1,
    "status": "ready",
    "requirement_parsed": {
      "platform": "xhs",
      "quantity_total": 10,
      "budget_min_cents": 0,
      "budget_max_cents": 3000000,
      "budget_raw": "3万以内"
    }
  },
  "error": null
}
```

`data.id` 是下游主链路使用的 `customer_demands` 主键。`demand_id/demand_version` 若存在，只作为需求版本事实保留。

需补充：

```json
{
  "success": true,
  "data": {
    "status": "draft",
    "missing_fields": ["submission_deadline_at"],
    "clarifying_questions": ["请确认提报截止时间"]
  },
  "error": null
}
```

### 3.4 Agent 停顿

- `status=draft`：停，问缺失项。
- `status=ready`：如果额外需求字段已匹配并确认，弹 `confirm-structured-brief`；确认后才 `search_creators`。

## 4. `search_creators`

### 4.1 输入

```json
{
  "id": "demand_123",
  "write_candidate_pool": true,
  "limit": 500
}
```

`id` 来自 `validate_requirement.data.id`，映射 `customer_demands` 主键，不代表新增需求表。

### 4.2 数据库动作

读：

- `customer_demands`
- `xhs_creator_accounts` / `dy_creator_accounts`

写：

- `creator_candidate_pool`

执行方式：

1. MCP 根据 `id` 读取 `customer_demands`。
2. 取需求表所有非空、已确认字段。
3. 按 `platform` 与 `xhs_creator_accounts` 或 `dy_creator_accounts` 同名字段匹配筛选。
4. 写入 `creator_candidate_pool` 后进入 MCN 排序。

### 4.3 输出

```json
{
  "success": true,
  "data": {
    "id": "candidate_pool_123",
    "requirement_id": "demand_123",
    "candidate_pool_written": true,
    "total_matched": 120,
    "supply_assessment": {
      "candidate_count": 120,
      "quantity_total": 10,
      "supply_multiplier": 12,
      "supply_risk_level": "low_risk",
      "should_start_manual_sourcing": false
    }
  },
  "error": null
}
```

## 5. `rank_mcns`

### 5.1 输入

```json
{
  "id": "candidate_pool_123"
}
```

中风险继续时：

```json
{
  "id": "candidate_pool_123",
  "medium_risk_confirmed": true
}
```

`medium_risk_confirmed` 只能在用户明确确认后设为 `true`。

### 5.2 数据库动作

读：

- `creator_candidate_pool`
- `mcn_agencies`

写：

- `mcn_recommendation_items`

执行方式：

1. MCP 根据 `id` 读取本轮 `creator_candidate_pool` 候选。
2. 从候选中提取相关 MCN/供应商。
3. 读取数据库中这些 MCN 的资料和供给内容。
4. 计算当前达人供需关系和建议手扒比例。
5. 拟出建议发送询价的 MCN 列表。

### 5.3 输出

```json
{
  "success": true,
  "data": {
    "id": "mcn_recommendation_123",
    "candidate_pool_id": "candidate_pool_123",
    "mcns": [
      {
        "mcn_id": "supplier_001",
        "agency_name": "机构A",
        "estimated_creator_count": 25,
        "mcn_rank_score": 86.5,
        "risk_notes": []
      }
    ],
    "inquiry_advice": {
      "selected_mcn_ids": ["supplier_001"],
      "cumulative_candidate_count": 120,
      "cumulative_supply_multiplier": 12,
      "should_continue_adding": false
    }
  },
  "error": null
}
```

### 5.4 Agent 停顿

`rank_mcns` 成功后，前端先展示当前达人供需关系、建议手扒比例和建议发送询价的 MCN 列表。用户可以同意或修改 MCN 列表。

用户确认 MCN 列表后，Agent 根据 `validate_requirement.data.id` 读取 `customer_demands` 非空字段，拟写企微消息，给用户确认。然后必须依次完成：

```text
confirm-supply-ratio
mcn-select-for-wechat
confirm-form-fields
confirm-wecom-permission
mcn-wechat-send
```

如果需要手扒，同步启动 `manual_source_creators` 对应的手扒程序，不等待 MCN 回填结束。

## 6. `create_with_distributions`

### 6.1 输入

`create_with_distributions` 的输入输出字段以企微发送接口为准，不能修改。业务需要的“每个 MCN 唯一填报链接”和“预填达人”通过接口已有能力承接：

- `supplierIds`：要发送的 MCN/供应商。
- `prefillRowsBySupplier` / `prefill_rows_by_supplier`：按 MCN/供应商分别预填候选达人。
- 响应里的 `distributions.created[].token` 或等价字段：每个 MCN 唯一填报链接/令牌。
- 不新增工具名，不恢复旧的 `create_mcn_inquiries`；底层 HTTP 端点仍是 `/api/projects/create-with-distributions/`。

预览：

```json
{
  "id": "mcn_recommendation_123",
  "projectName": "美妆达人提报",
  "description": "请在截止时间前回填可提报达人、报价、返点和档期。",
  "deadline": "2026-07-10T18:00:00+08:00",
  "remindAt": "2026-07-10T12:00:00+08:00",
  "usageScope": "project",
  "supplierIds": ["supplier_001"],
  "sendWechatNotification": true,
  "prefillRowsBySupplier": {
    "supplier_001": [
      {"talentName": "达人A", "price": 200000, "rebate": 0.2}
    ]
  },
  "preview_only": true
}
```

正式发送：

```json
{
  "id": "mcn_recommendation_123",
  "projectName": "美妆达人提报",
  "description": "请在截止时间前回填可提报达人、报价、返点和档期。",
  "deadline": "2026-07-10T18:00:00+08:00",
  "remindAt": "2026-07-10T12:00:00+08:00",
  "usageScope": "project",
  "supplierIds": ["supplier_001"],
  "sendWechatNotification": true,
  "prefillRowsBySupplier": {
    "supplier_001": [
      {"talentName": "达人A", "price": 200000, "rebate": 0.2}
    ]
  },
  "preview_only": false
}
```

`prefillRowsBySupplier` 只能包含候选池中属于对应 MCN/供应商的达人，作为推荐底稿。Agent 不自己生成填报链接。

### 6.2 Hooks 门控

调用前会检查：

| 检查 | 当前行为 |
|---|---|
| 工具名 | 必须是 `create_with_distributions` |
| 禁止直连 | Bash/PowerShell/curl 调接口会被阻断 |
| 时间 | `deadline/remindAt` 必须未来、带时区 ISO |
| 供应商 | `supplierIds/supplier_ids` 非空字符串数组 |
| `usageScope` | 缺失补 `project`，传错阻断 |
| 角色 | 若传入角色，只允许 `media/procurement` |
| 前置 gate | 发送前确认未完成会阻断 |

### 6.3 数据库动作

读：

- `mcn_recommendation_items`
- `mcn_agencies`
- `creator_candidate_pool`

写：

- 后端项目/分发记录
- 必要时关联 `mcn_inquiries`

### 6.4 输出

```json
{
  "success": true,
  "data": {
    "id": "distribution_123",
    "inquiry_ids": ["inquiry_001"],
    "status": "sent",
    "distributions": {
      "created": [
        {
          "supplier": "supplier_001",
          "token": "public-form-token",
          "status": "pending"
        }
      ]
    }
  },
  "error": null
}
```

正式发送成功后，Hooks 设置等待锁。Agent 停下来等待机构回填和手扒结果，不能直接继续精排。

## 7. `ingest_mcn_submissions`

### 7.1 输入

```json
{
  "inquiry_id": "inquiry_001",
  "items": [
    {
      "platform": "xhs",
      "platform_account_id": "account_001",
      "nickname": "达人A",
      "price_cents": 200000,
      "rebate_rate": 0.2,
      "schedule_info": "7月可排"
    }
  ]
}
```

`items` 必须来自真实 MCN 回填，不得由 Agent 补造。机构通过各自唯一填报链接回填后，导入工具把回填达人回收到候选池。

### 7.2 数据库动作

读：

- `mcn_inquiries`
- 真实回填源

写：

- `mcn_submission_items`
- `creator_supply_offers`
- `creator_candidate_pool`，`candidate_source=mcn_returned`
- 可更新 `xhs_creator_accounts` / `dy_creator_accounts`

### 7.3 输出

```json
{
  "success": true,
  "data": {
    "id": "mcn_ingest_batch_123",
    "accepted_count": 8,
    "rejected_count": 2,
    "created_candidate_count": 8
  },
  "error": null
}
```

按运行时 schema 使用当前需求/候选上下文 `id`。手扒在判断需要补量时同步启动，不等 MCN 回填结束：

```json
{
  "id": "candidate_pool_123",
  "manual_results": [
    {
      "platform": "xhs",
      "platform_account_id": "account_002",
      "nickname": "达人B",
      "profile_url": "https://..."
    }
  ]
}
```

### 8.2 数据库动作

写：

- `creator_candidate_pool`，`candidate_source=manual_sourced`
- 可写 `creator_supply_offers`
- 可更新 `xhs_creator_accounts` / `dy_creator_accounts`

### 8.3 输出

```json
{
  "success": true,
  "data": {
    "id": "manual_source_batch_123",
    "imported_count": 5,
    "created_candidate_count": 5
  },
  "error": null
}
```

## 9. `rank_creators`

### 9.1 输入

以运行时 schema 为准。当前 Skill 期望至少：

```json
{
  "id": "candidate_pool_123",
  "ranking_strategy": "default"
}
```

`id` 取回填/手扒汇总后的候选池或当前运行时 schema 要求的上游结果 ID。不能在企微分发未完成、用户未确认精排时调用。

精排前必须已经回收机构回填和手扒结果。`rank_creators` 对库内候选、MCN 回填达人和手扒达人统一去重、筛选、排序。

### 9.2 Hooks 门控

会阻断：

- 没有 `create_with_distributions` 成功证据。
- 分发成功后还没有用户继续确认。
- `workflow_state.allowed_actions` 不允许。
- 平台 `mcn_phase` 未达 `ingested` 且不是 `not_required`。
- `high_risk` 未恢复。

### 9.3 数据库动作

读：

- `customer_demands`
- `creator_candidate_pool`

写：

- `recommendation_runs`
- `creator_recommendation_items`

### 9.4 输出

```json
{
  "success": true,
  "data": {
    "run_id": "run_123",
    "ranked_count": 30,
    "dedupe_summary": {
      "before_count": 45,
      "after_count": 30
    },
    "items": []
  },
  "error": null
}
```

## 10. `create_submission_batch`

### 10.1 输入

```json
{
  "run_id": "run_123"
}
```

风险账号已确认时：

```json
{
  "run_id": "run_123",
  "allow_need_confirm_with_risk": true
}
```

### 10.2 数据库动作

读：

- `recommendation_runs`

写：

- `creator_submissions`

### 10.3 输出

```json
{
  "success": true,
  "data": {
    "id": "submission_batch_123",
    "batch_no": 1,

```

## 11. `record_client_feedback`

### 11.1 输入

当前工具文档以运行时 schema 为准，目标形态：

```json
{
  "run_id": "run_123",
  "feedback_items": [
    {
      "submission_id": "submission_001",
      "status": "rejected",
      "reason": "价格偏高"
    }
  ]
}
```

如果线上 schema 仍是 `submission_batch_id + feedback_type + feedback_content`，Agent 按 schema 传，不强行塞目标字段。

### 11.2 数据库动作

写：

- `creator_submissions` 反馈字段
- 需求核心变更时创建 `customer_demands` 新版本

### 11.3 输出

```json
{
  "success": true,
  "data": {
    "updated_count": 1,
    "next_action": "continue_submission"
  },
  "error": null
}
```

## 12. 错误处理

| 情况 | Agent 行为 |
|---|---|
| 工具不存在 | 停，报 `integration_required` |
| schema 缺字段/冲突 | 停，按运行时 schema 说明冲突 |
| `success=false` | 停，展示业务错误和下一步 |
| 缺 `trace_id` | 不阻断；需要排障再提醒后端查 |
| 写调用超时 | 不盲目重试，先查业务 ID 或让后端排查 |
| ID 来源不明 | 停，不构造 ID |
